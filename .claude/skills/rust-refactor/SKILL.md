# Rust Refactor (climem)

Безопасный рефакторинг этого Rust-проекта с предварительным анализом влияния.
Импакт-анализ берётся из **карты кода** проекта — `memory\cm.exe map` — а НЕ из
внешнего LSP/rust-analyzer (его в этом окружении нет на PATH). Карта знает границы
символов (не ловит имя в строке/комменте), различает определение и вызов и
отвечает за один вызов — ровно то, что нужно, чтобы переименование/перемещение не
пропустило ни одной точки использования.

Контракт `cm map`: `memory\cm.exe help`. Карта проекта уже построена; после правок
её надо пере-индексировать: `memory\cm.exe map ./src` (инкрементально).

## Использование

```
/rust-refactor <action> <target> [--dry-run]
```

**Действия:**
- `rename <old> <new>` — переименовать символ
- `extract-fn <file>:<start>-<end>` — вынести выделенный блок в функцию
- `inline <fn>` — заинлайнить функцию
- `move <symbol> <dest-module>` — перенести символ в другой модуль

**Примеры:**
- `/rust-refactor rename parse_config load_config`
- `/rust-refactor extract-fn src/main.rs:20-35`
- `/rust-refactor move UserService src/services/`

По умолчанию работай как `--dry-run`: сперва покажи таблицу влияния, и только после
явного подтверждения применяй правки. `--dry-run` означает «только превью».

## Чем заменяем LSP-операции

| Нужно (в оригинале — LSP) | В этом проекте |
|---|---|
| `findReferences` (все использования) | `memory\cm.exe map --uses <name>` |
| `goToDefinition` / `hover` (где определён) | `memory\cm.exe map --query <name>` |
| исходящие зависимости символа | `memory\cm.exe map --calls <name>` |
| что определяет файл/модуль | `memory\cm.exe map --defines <file>` |
| есть ли похожий символ / конфликт имени | `memory\cm.exe map --query <new>` и `--like <substr>` |

Каждая строка вывода `cm map` — JSON.
- `--query`/`--like`/`--defines`/`--list` → `{"name","kind","path","line","signature"}`
- `--uses` → `{"name","kind","path","line","def_line"}` (`line` = точка использования,
  `name`/`def_line` — вызывающий символ)
- `--calls` → `{"calls","line","resolved"}` (только внутрипроектные; `--external`
  добавит stdlib/3rd-party)

**Тесты по умолчанию скрыты** — добавляй `--tests`, чтобы импакт-анализ покрыл и
`tests/`-файлы, и inline `mod tests`. Без этого ты пропустишь тестовые вызовы.

## ⚠️ Когда карте нельзя доверять (тогда — grep)

Это не опционально, а условие безопасности рефакторинга:

- **Общие/перегруженные имена** (`new`, `get`, `run`, `handle`, или имя, определённое
  в нескольких файлах): `--uses`/`--calls` резолвят ПО ИМЕНИ без анализа областей
  видимости, поэтому сливают одноимённые символы и могут как пропустить, так и
  приписать лишнее. Для таких имён проверяй `--uses` дополнительным `Grep` и читай
  каждую точку.
- **Свободный текст, не символ**: имя в строковом литерале, комментарии, doc-комменте,
  ключ в `.md`/`.json`/`.toml`, имя в макросе — карта видит только символы кода. Для
  doc-комментариев и `help::HELP`/README используй `Grep` (см. инвариант проекта:
  контракт и help держатся в синхроне).
- **Только что отредактированный код**: карта — это кэш. После каждой правки гоняй
  `memory\cm.exe map ./src` заново, иначе следующий шаг анализа врёт.

Если имя уникально по проекту — карты достаточно. Иначе обязательно подстрахуйся grep'ом.

## Воркфлоу

### 1. Rename Symbol

1. `cm map --query <old>` — найти определение (где, какой `kind`, сигнатура). Если
   определений несколько — имя НЕ уникально, переходи в режим grep (см. выше).
2. `cm map --uses <old>` (с `--tests`) — все точки использования, каждая с вызывающей
   функцией и строкой.
3. `cm map --query <new>` и `cm map --like <new>` — проверить конфликт: не занято ли
   новое имя.
4. `Grep` по `<old>` в `.md`/комментариях/`help::HELP`/README — карта их не покажет.
5. Собрать таблицу влияния (см. формат ниже). В `--dry-run` остановиться здесь.
6. После подтверждения — `Edit` по каждой точке.
7. `cm map ./src` — пере-индексировать. Повторно `cm map --uses <old>` должен вернуть
   пусто (старого имени не осталось).
8. `cargo check` (через `~/.cargo/bin`, см. ниже) — убедиться, что компилируется.

**Формат вывода (dry-run):**

```
## Rename: parse_config → load_config

### Импакт-анализ
**Определение:** src/config.rs:25 — `pub fn parse_config(`
**Использований найдено (cm map --uses):** N

| Файл | Строка | Вызывающий | Тип |
|------|--------|-----------|-----|
| config.rs | 45 | save_config (def 40) | вызов |
| main.rs   | 30 | main (def 10)       | вызов |
| ...       | ... | ...                | ...  |

### Потенциальные проблемы
⚠️ Публичный API: src/lib.rs — `pub use` / `pub fn` → ломает внешних потребителей
⚠️ Документация: grep нашёл `parse_config` в README.md / doc-комментах — обновить вручную
⚠️ Конфликт: `cm map --query load_config` → имя уже занято? (если да — стоп)

### Применить? (да / только превью)
```

### 2. Extract Function

1. `Read` блок `<file>:<start>-<end>`.
2. Анализ переменных: входы (используются, но не объявлены в блоке) → параметры;
   выходы (объявлены в блоке, используются после) → возвращаемое значение; локальные
   (объявлены и используются только в блоке) → остаются внутри.
3. Учесть ранние `return`, `?`, `break`/`continue`, заимствования (`&`/`&mut`),
   `async`/`.await`. Сигнатуру выбрать под `Result<…>`, если в блоке есть `?`.
4. Сгенерировать функцию + заменить блок вызовом.
5. `cargo check` — компиляция.

Соблюдай стиль проекта: модуль открывается `//!` doc-комментом; комментарии и
сигнатуры — на английском (см. CLAUDE.md).

### 3. Inline Function

1. `cm map --uses <fn>` (с `--tests`) — все вызовы.
2. `cm map --query <fn>` — тело функции (`Read` по `path:line`).
3. Подставить тело в каждую точку вызова, согласовав имена параметров с аргументами.
4. Удалить определение. `cm map ./src` + `cargo check`.

### 4. Move Symbol

1. `cm map --defines <src-file>` — что определено рядом (возможные со-зависимости,
   которые поедут вместе).
2. `cm map --uses <symbol>` (с `--tests`) — все потребители → им нужно поправить `use`.
3. `cm map --calls <symbol>` — на что символ опирается; не возникнет ли циклическая
   зависимость между старым и новым модулем.
4. План: новый файл (+ `pub mod …;` в `mod.rs`/родителе), удаление из старого,
   правка `use`-путей у каждого потребителя.
5. Применить → `cm map ./src` → `cargo check`.

**Формат вывода (dry-run) для move:**

```
## Move: UserService → src/services/user.rs

### Текущее место
src/handlers/auth.rs (cm map --defines)

### Поедут вместе (со-зависимости)
- struct UserService
- impl UserService

### Правки use (cm map --uses)
| Файл | Сейчас | Станет |
|------|--------|--------|
| main.rs | use handlers::auth::UserService | use services::user::UserService |

### Проверка циклов (cm map --calls)
✅ / ⚠️ <вывод>
```

## Проверки безопасности

| Проверка | Команда / способ |
|---|---|
| Полнота ссылок | `cm map --uses <sym> --tests` + grep для общих имён |
| Конфликт имён | `cm map --query <new>` / `cm map --like <new>` |
| Смена видимости | прочитать сигнатуру (`signature` в выводе): `pub` → внешний API |
| Код в макросах | grep — карта не видит сгенерированное макросом |
| Документация | grep по `.md` и doc-комментам (карта их не индексирует) |
| Покрытие тестами | те же запросы с `--tests` |
| Компиляция | `cargo check` после правок |

## Запуск cargo / пере-индексация

`cargo`/`rustc`/`rust-analyzer` установлены, но НЕ на PATH — сперва добавь
`~/.cargo/bin`:

```bash
export PATH="$HOME/.cargo/bin:$PATH"
cargo check                 # быстрая проверка после правок
```

Сборка тянет SQLite из исходников (`rusqlite` bundled) — нужен C-тулчейн (MSVC на
Windows). После любого изменения исходников обязательно:

```
memory\cm.exe map ./src     # инкрементальная пере-индексация карты кода
```

иначе последующие шаги анализа будут опираться на устаревший кэш.

## Инварианты проекта, которые рефакторинг легко ломает

Перед применением сверься с CLAUDE.md / `memory/notes/decisions.md`. Особенно при
переименовании/перемещении публичных штук:

- Переименование команды/флага/формата → синхронно правь `help::HELP`, `cli::VALUE_FLAGS`
  и README.md (контракт живёт в бинарнике).
- Новый флаг со значением → добавь в `cli::VALUE_FLAGS`, иначе `--flag value`
  распарсится как bool-флаг + позиционный аргумент.
- Любой путь вставки/удаления заметки трогает и `notes`, и `notes_fts` (они
  синхронизируются вручную).
- Не добавляй конфиг-поле под сырой ключ-секрет; секреты только из окружения.

# Грабли и соглашения (gotchas)

Неинтуитивные зависимости и ловушки, которые легко переоткрыть. То, что очевидно из кода, —
не дублируем.

## Контракт CLI

- **stdin на Windows / PowerShell 5.1: задай кодировку явно.** PowerShell 5.1 по умолчанию
  пайпит UTF-16 LE (кириллица → `???`) и даже при `$OutputEncoding = [System.Text.Encoding]::UTF8`
  вставляет **два** BOM (EF BB BF) перед текстом — один как preamble потока, второй на каждую
  строку. `read_stdin` (commands.rs) стрипит их через `trim_start_matches('\u{feff}')`. Для
  пользователя: задай **один раз** в профиле `$OutputEncoding = [System.Text.UTF8Encoding]::new($false)` (no-BOM).
  `[System.Text.Encoding]::UTF8` тоже работает — BOM уберёт бинарник.
- **`init` читает интерактивный y/n из stdin** (функция `init::prompt_yes_no`). Это единственное
  исключение из правила «stdin = тело заметки»: `init` сам не принимает тело заметки, поэтому
  конфликта нет. При пайпе stdin EOF → `false` (тихий пропуск, скрипты не ломаются).
- **Тело заметки — всегда через stdin**, не аргументом (`commands::remember` → `read_stdin`).
  Это снимает экранирование кавычек/многострочности. Пустой stdin → ошибка с подсказкой.
  Правило держится во всех новых командах, что принимают произвольный текст.
- **Вывод — JSONL** (по объекту на строку, `output::print_line`). Не печатай человекочитаемый
  текст в stdout у команд данных — он ломает парсинг вызывающей стороной. Исключения, которые
  пишут pretty-JSON/текст: `config` (показ), `export` без `--out`, `help`.
- **`recall` печатает «худую» строку** (`output::recall_value`, плана E1): по умолчанию только
  `id, kind, body` + `tags`/`origin`/`source` (если не пусты/не null). Отладочные `score/fts/
  vector` — под `--explain`; `--fields a,b,c` — ровно перечисленные поля (whitelist
  `output::RECALL_FIELDS`). Полная запись — всегда `get <id>`. Меняешь набор полей — правь и
  `RECALL_FIELDS`, и `help::HELP`, и `README`.
- **`recall` — preview-first по умолчанию (бюджет тела, заимствовано из graphify):**
  `output::insert_body` сравнивает длину тела с per-result бюджетом. В пределах бюджета тело
  идёт целиком под `body` (короткие `remember`-факты — байт-в-байт как раньше, без лишних
  `get`-раундтрипов). Сверх бюджета — `body` ЗАМЕНЯЕТСЯ на `preview` (обрезка+`…`) + `chars`
  (полная длина), чтобы агент дотянул остальное `cm get <id>`. Смена ключа `body→preview`
  намеренна: тот, кто читает `body`, замечает его отсутствие, а не действует на обрезке. Дефолт
  бюджета — `search.recall_body_chars` (500); `recall --budget N` перебивает на вызов
  (`commands::parse_budget`), `--budget 0`/`--full` снимают обрезку (тело целиком). С `--fields`
  бюджет НЕ применяется (явная проекция: `body` = целиком), предупреждение в stderr.
- **Ключи JSON выводятся в алфавитном порядке** (serde_json без фичи `preserve_order` →
  `Map` = `BTreeMap`). Это намеренно: байт-стабильный вывод — кэшируемый префикс для
  потребляющего агента (плана A2). Не закладывайся на порядок вставки.
- **Подсказки и нарратив — в stderr.** `init` печатает указатель в stderr, данные — в stdout.
  Ошибки (`AppError`) — тоже stderr, с блоком `пример:` если задан `with_hint`.
- **Два стиля вызова**: подкоманды (`cm recall …`) и Windows `/флаг` (`cm /recall …`).
  В `/`-стиле подкоманда — только первый токен; дальнейшие `/x` парсятся как флаги
  (`cli::Parsed::parse`).
- **Новый value-флаг надо зарегистрировать** в `cli::VALUE_FLAGS`, иначе `--flag value`
  распарсится как булев флаг + позиционный аргумент. `--flag=value` работает всегда.
- **Позиционный путь с ведущим `/` ломается.** Любой токен, начинающийся с `/` (кроме первого
  = подкоманды), `cli::Parsed::parse` трактует как Windows-флаг. Значит **абсолютный
  Unix-путь** как позиционный аргумент (`cm import /abs/file.md`) не доходит до команды —
  `arg(0)` пуст → «import needs a file path». Абсолютный Windows-путь (`D:\…`) начинается с
  буквы диска и работает. В тестах поэтому позиционные пути — относительные (без ведущего `/`),
  с управлением cwd (`tests/common::run_cwd`). Кросс-платформенная заноза, не баг конкретной
  команды.

## Эмбеддинги и поиск

- **Дрейф эмбеддера.** Сменил `provider`/`model`/`dimension` на непустом хранилище — старые
  векторы не сойдутся с новыми. `commands::warn_on_drift` сверяет `embedder_signature` в
  `meta` и печатает **нефатальное** предупреждение. Лечение: переимпорт или новое хранилище.
- **Несовпадение размерности не падает**: `embed::cosine` возвращает 0 при разной длине
  векторов — поиск деградирует мягко, а не паникует.
- **FTS5 не делает стемминг.** Запрос «авторизация» не найдёт «авторизацию». Поэтому
  существует векторный канал на char-3grams (`embed/hashing.rs`) — ловит общую морфему. Не
  «чини» это, добавляя стемминг в FTS; гибрид — осознанное решение (см. `desc.md`,
  [notes/decisions.md](notes/decisions.md)).
- **`chunking.max_tokens` считается в СЛОВАХ, не BPE-токенах** (`chunk::{markdown,text}` через
  `split_whitespace`). Имя оставлено ради совместимости конфига; ~200 слов ≈ 256 токенов
  (дефолт снижен в плане C1). Сменил дефолт/размер — **переимпортируй** (старые чанки хранят
  прежний размер).
- **Пиши доки под лексический индекс (плана A1).** Ретрив токенизирует ровно то, что в него
  кладут. Делай каждый заголовок самодостаточным извлекаемым фактом; называй канонический
  термин (+ ключевые синонимы/акронимы один раз), чтобы поверхностное совпадение срабатывало;
  один факт — одно место + ссылки. Тогда хватит меньшего `--limit`. Импорт **идемпотентен
  per-original** (повторный `import` того же файла заменяет его чанки), но **кросс-документного
  дедупа нет** — один и тот же текст в разных файлах раздувает стор (ср. [roadmap.md](roadmap.md) X3).
- **Локальный эмбеддер — не семантика, а лексическая/морфологическая близость.** Знаковое
  FNV-1a хеширование (`fnv1a`) + L2-норма. Для настоящей семантики — провайдер `api`.
- **Слияние каналов — RRF, не min-max.** `score = w_fts/(k+rank_fts) + w_vec/(k+rank_vec)`
  (`search::recall_with`, `k = search.rrf_k`). Числа **мелкие** (`~w/(k+rank)`, не [0,1]!) и
  `score == fts + vector`. Старое правило «score всегда в [0,1]» больше не действует — `--min-
  score` задаётся в этой мелкой шкале. Решение и почему — [notes/decisions.md](notes/decisions.md).
- **Порядок `recall` детерминирован**: кандидаты из `BTreeSet`, финальная сортировка с
  tie-break по `id`. Не вноси `HashMap`-итерацию в путь ранжирования (был баг недетерминизма).
- **`recall` сужается метаданными до скоринга**: `--tag`/`--origin-prefix` → `store::ids_matching`
  (плана R4). Над-фильтрация (плохие теги) может выкинуть релевантный док — это by design.

## Секреты

- **Ключ API никогда не в `config.json`.** В конфиге только *имя* env-переменной
  (`embedding.api_key_env`, дефолт `MEMORY_EMBED_API_KEY`); сам ключ читается из окружения
  (`ApiEmbedder::from_config`). Не добавляй поле под сырой ключ.
- `config::mask_secrets` маскирует значения ключей, содержащих `key`/`secret`/`token`/
  `password`, **кроме** `api_key_env` (там лишь имя переменной). Учитывай при добавлении полей.
- **Маскирование работает только на объекте, не на листе.** `mask_secrets` инспектирует
  *имена ключей*; полному показу `config` оно видит `api_key` и маскирует. Но `config get
  embedding.api_key` и эхо `config set …api_key` передают в `mask_secrets` **сам строковый
  лист** (ключа рядом нет) → значение возвращается **сырым**, не `***`. Это утечка секрета в
  выводе get/set — зафиксирована тестами как regression-anchor (см.
  [known-issues.md](known-issues.md)).

## Хранилище

- **`store.db` — производный и одноразовый** (`desc.md §3`, решение в
  [notes/decisions.md](notes/decisions.md)). Правда — `notes/<id>.md` + `imports/`
  (оригиналы + `.meta.json`-сайдкары). **Удалить `store.db` безопасно** — `cm reindex`
  пересоберёт всё из файлов (`Store::open` сам создаёт файл). Не клади в `store.db` ничего, что
  не выводится из md/сайдкаров, иначе появятся два источника правды.
- **`remember` пишет md ПЕРВЫМ, потом индексирует best-effort.** `notes/<id>.md` — точка
  коммита; кривой эмбеддер даёт предупреждение в stderr, но заметка сохранена и `{"id"}`
  напечатан (`commands::index_note_best_effort`). Любая новая команда со свободным текстом —
  так же: сперва файл-правда, потом индекс.
- **`forget` удаляет md + строку индекса; чанки удалить нельзя.** `kind=chunk` (производный из
  `imports/`) → отказ с подсказкой «правь imports/ + reindex». Не «чини» это удалением строки
  индекса — следующий `reindex` её вернёт.
- **Прямые правки md видит только `reindex`** (по `content_hash` в таблице `sync`). Правка
  файла мимо инструмента не попадёт в поиск/граф до пересборки — это by design, не баг.
- **`reindex` обнуляет epoch `created_at`, сохраняя только `created_iso`.** `commands::reindex_notes`
  передаёт в `upsert_note` `created_at = 0` (восстанавливает лишь ISO-строку из frontmatter `created`).
  Поэтому **сортировать заметки по свежести надо по строке `created_iso`, не по i64 `created_at`** —
  иначе после любого `reindex` ключ схлопывается в 0 и порядок падает на случайный tie-break по
  hex-id (id из `mint_hex` нехронологичны). ISO-8601 `YYYY-MM-DDTHH:MM:SSZ` сортируется
  лексикографически = хронологически. Поймано ревью на `commands::feedback_list` (см.
  [notes/decisions.md](notes/decisions.md) «cm feedback»).
- **`deinit` — полный откат `init`, точность зависит от манифеста.** `init` пишет
  `<data>/.init-manifest.json` (исходные пути импортированных доков, до-init `.gitignore`,
  создавался ли `AGENTS.md`); `deinit` по нему восстанавливает доки на исходные пути, чинит
  `.gitignore` и сносит папку памяти целиком. Путь дока хранится ОТНОСИТЕЛЬНО target. Если после
  init двигать/удалять оригиналы или править `memory/` руками (или манифеста нет — старый store),
  откат деградирует до best-effort: доки восстанавливаются по basename из сайдкара в
  `docs/climem/`, из `.gitignore` снимается только cm-блок по `GITIGNORE_MARKER`. Конфликт с
  существующим файлом юзера → `<dir>/climem/<file>`, не затирая.
- **`reindex --all` стирает производное, `oplog`/`meta` переживают.** `wipe_derived` чистит
  `notes`/`notes_fts`/`sync`/`imports`/`edges`; история операций и `embedder_signature`
  остаются (не реконструируемы). Есть legacy-guard: `--all` откажет, если в БД есть заметки, а
  `notes/` пуст (затёр бы старые данные). При недоступном эмбеддере `reindex` деградирует до
  FTS-only (keyword-индекс + sync, без векторов).
- **Rollback-журнал, не WAL** — чтобы `store.db` оставался одним файлом. `.gitignore`
  (и тот, что `init` пишет В папку памяти) исключает `*.db-wal`/`*.db-shm`. Не включай WAL.
- **Теперь когерентными надо держать ТРИ вещи: md, `notes`, `notes_fts`.** `notes_fts` —
  обычная (не `content=`) FTS5: `upsert_note` делает DELETE+INSERT по rowid, `forget`/
  `delete_chunks_for_source` — удаление. Добавляешь путь записи/удаления заметок — тронь обе
  таблицы И помни, что правда живёт в md (отрази изменение там же).
- **Граф рёбер (`edges`) выводится из md массово** (`commands::index_note_edges` через
  `graph::note_edges`), не правится точечно: `remember` derives рёбра одной заметки, `reindex`
  — авторитетный полный проход. Висячая цель (`dst_id NULL`) — нормально, оживает позже
  (инкрементальный `reindex` оживляет её даже у НЕизменившейся заметки — инвариант (i)).
- **Любой путь, удаляющий/прунящий заметку, обязан тронуть И входящие рёбра**, не только
  исходящие: рядом с `delete_edges_from(id)` (исходящие) зови `dangle_edges_to(id)` (входящие →
  висячие). Иначе остаётся «осиротевшее разрешённое ребро» на мёртвый id (инвариант (h)).
- **Разрешение цели графа — slug по умолчанию, префикс `id:` форсирует id**
  (`graph::resolve_target`), **без hex-эвристики** (slug `db-schema` не спутается с id).
  `normalize_slug` схлопывает ` `/`-`/`_`→`-` и lowercase'ит, но **кириллицу сохраняет** (в
  отличие от FTS — диакритику/алфавит не режет). При коллизии slug побеждает наименьший id
  (`remember` и `reindex` оба предупреждают на stderr).
- **Предикат рёбер нормализуется** через `graph::normalize_predicate` (lowercase + схлопывание
  ` `/`-`/`_`→`_`, но НЕ camelCase): `depends_on`≡`depends-on`≡`Depends On`. Применяется в
  `note_edges` и к фильтру `--predicate` (`related`/`backlinks`/`graph_predicate`) — стиль
  предиката неважен. Стиль `dst_raw` важен (часть PK), стиль предиката — нет.
- **Граф направленный.** `A→B` = «A ссылается/зависит от B». Вперёд — `related` (`edges_from`),
  назад — `backlinks` (`edges_to`, индекс `edges_dst`). Граф-обход — одно ядро
  `search::bfs_neighbors` (live-validate через `store.get` в обоих трактах). Рёбра дедупятся по
  `(predicate, resolved-key)` в `index_note_edges` до вставки (PK по `dst_raw` не трогаем).
- **Граф-канал `recall --related`** off по умолчанию (`hybrid_weights.graph==0`) — при `--related`
  с нулевым весом печатается stderr-хинт. Глубина и предикат канала — в конфиге
  (`search.graph_depth` дефолт 2, `search.graph_predicate`), а не хардкод. `cm related` ходит на
  свою глубину (`--depth`, дефолт 1) — команды намеренно разные.
- **id заметки — короткий lowercase-hex** (`store::mint_hex`), авторится в md, `id ==
  notes/<id>.md`. `commands::parse_id` принимает только `^[0-9a-f]+$`. Не возвращай к
  autoincrement: id должен пережить пересборку и не зависеть от порядка вставки.
- `busy_timeout = 5s` (`Store::open`) — на случай параллельных короткоживущих процессов.

## Время и зависимости

- **Время считается без crate'ов** (`util::iso_utc` + `civil_from_days`, алгоритм Хиннанта).
  Не тащи `chrono`/`time` ради форматирования дат — это сознательно минимальное дерево
  зависимостей.

## Тулчейн

- `cargo`/`rustc` могут быть **не на PATH** — добавь `~/.cargo/bin` перед сборкой (см.
  [build.md](build.md) и авто-память `rust-toolchain-path`).

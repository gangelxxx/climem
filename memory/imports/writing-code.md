# Процесс написания кода по запросу

Порядок, который держит изменения дешёвыми и не ломает инварианты.

1. **Сориентируйся по памяти.** Прочитай [../overview.md](../overview.md) и
   [../structure.md](../structure.md); для затрагиваемой области — [../conventions.md](../conventions.md)
   и [../notes/decisions.md](../notes/decisions.md). Есть готовый рецепт в
   [../howto/](../howto/)? Следуй ему.
2. **Уважай инварианты** из [../notes/decisions.md](../notes/decisions.md): короткоживущий
   процесс без фонового состояния; **md — правда, `store.db` — производный одноразовый индекс**
   (запись файла — точка коммита, индексация best-effort, всё выразимо в md, чанки
   контент-адресны и не удаляются точечно); `store.db` — один файл (rollback-журнал, не WAL);
   гибрид скрыт за `recall`; секреты только из окружения; `help` = контракт; тело через
   stdin, вывод JSONL. Хочешь нарушить — сначала зафиксируй новое решение в `decisions.md`.
3. **План для нетривиального.** Затрагивает несколько модулей/схему/контракт — набросай
   шаги (можно в `analysis/<topic>-plan.md`, если крупно).
4. **Пиши в стиле окружения** — см. [code-style.md](code-style.md). Минимизируй новые
   зависимости (направление проекта — тонкое дерево депов).
5. **Контракт и помощь синхронно.** Меняешь команду/флаг/формат — правь `help::HELP`,
   `cli::VALUE_FLAGS` и (если для пользователя) корневой `../README.md` в **том же** изменении.
6. **Проверь**: `cargo build --release` (+`--features pdf`/нужная фича), `cargo clippy`,
   ручной прогон на scratch-папке `--dir ./.testmem`. Тесты — `cargo test`, если есть.
7. **Обнови память** (это часть работы, не долг): задетые из
   [../overview.md](../overview.md)/[../structure.md](../structure.md)/[../build.md](../build.md)/
   [../conventions.md](../conventions.md); баг → [../known-issues.md](../known-issues.md);
   решение → `decisions.md`; в конце сессии — запись в [../journal.md](../journal.md)
   (что и **почему**).

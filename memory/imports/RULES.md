# Свод правил ведения памяти climem

Единственный авторитетный регламент по работе с `memory/`. [README.md](README.md) — только
индекс; правила живут здесь.

Память бесполезна, если устаревает или дублирует код. Эти правила держат её **тонкой,
точной и живой**.

> Это регламент про **ведение памяти**. Правила про **работу над кодом climem** (написание
> по запросу, оформление, ревью) — в [rules/](rules/). Их легко перепутать.

## 1. Назначение

`memory/` — **версионируемый источник правды** о проекте: структура, сборка, решения, баги,
рецепты, журнал. Цель — чтобы новая сессия (модель или человек) за минуты поняла проект и
могла **быстро починить баг или внедрить фичу**, не переоткрывая известное.

## 2. Карта: что куда писать

| Тип знания | Файл |
|------------|------|
| Что за проект, стек, версии, фичи сборки, активная задача | [overview.md](overview.md) |
| Карта модулей `src/`, модель данных (`notes/`+`imports/` = правда, `store.db` = производный индекс + граф), потоки recall/reindex/import/embed | [structure.md](structure.md) |
| Сборка, фичи cargo, профиль, тулчейн, дистрибуция | [build.md](build.md) |
| Баги: открытые и закрытые | [known-issues.md](known-issues.md) |
| Грабли, gotchas, неинтуитивные зависимости | [conventions.md](conventions.md) |
| Пошаговые рецепты «как сделать X» | [howto/](howto/) |
| Архитектурные решения и инварианты | [notes/decisions.md](notes/decisions.md) |
| Заметки по конкретным фичам/компонентам | [notes/](notes/) |
| Глубокие разборы и планы по задачам | `analysis/` (завести по событию) |
| Фичи/точки расширения в планах | [roadmap.md](roadmap.md) |
| Журнал сессий: что сделано и почему | [journal.md](journal.md) |

## 3. Когда обновлять (триггеры — обязательно)

Обновляй память **в том же изменении, что и код**, не «потом»:

1. Поменял команды сборки/фичи/профиль → [build.md](build.md).
2. Добавил/удалил dependency или модуль `src/`, изменил схему SQLite → [structure.md](structure.md).
3. Бампнул версию crate, изменил версию схемы (`config.version` / `SCHEMA`), сменил
   активную задачу → [overview.md](overview.md).
4. Нашёл баг → «Открытые» в [known-issues.md](known-issues.md). Починил → перенеси в
   «Закрытые» с «как починили» + ссылка на коммит/`analysis/`.
5. Принял неочевидное архитектурное решение → [notes/decisions.md](notes/decisions.md).
6. Напоролся на грабли → [conventions.md](conventions.md).
7. Запланировал/реализовал точку расширения (MCP, нейромодель, формат) → [roadmap.md](roadmap.md).
8. Разобрался в нетривиальной области надолго → [notes/](notes/), а если это глубокий
   разбор/план — заведи `analysis/<topic>.md`.
9. Завершаешь сессию → запись в [journal.md](journal.md) (что сделано, **почему**, что осталось).

## 4. Как писать

- Кратко, по делу, плотный технический русский — в стиле существующих доков и `desc.md`.
  Комментарии в **коде** — только по-английски (как уже принято в `src/`).
- **Один факт — одно место.** Не дублируй между файлами — **ссылайся** (`[текст](путь)`).
- Версии crate'ов / версия схемы — единственный источник в [overview.md](overview.md).
- Даты — **абсолютные** (`2026-06-17`), не «вчера/недавно».
- Ссылайся на код по **пути + имени символа** (функция/тип/модуль), не по номеру строки —
  строки уезжают. Номер можно как «≈»-подсказку.
- Перед добавлением проверь, нет ли уже файла/раздела про это — **обнови**, не плоди дубль.

## 5. Жизненный цикл знания и рост

```
планирую расширение → roadmap.md (строка) / analysis/<topic>-plan.md (если план большой)
делаю               → notes/<тема>.md по ходу
разобрался глубоко  → analysis/<topic>.md
принял решение      → notes/decisions.md
нашёл/починил баг   → known-issues.md (Открытые → Закрытые)
завершил сессию     → journal.md
```

**Файл заводится, когда есть что в него положить**, не «про запас». Память — не каркас ради
полноты. Подпапку (`analysis/`, новые в `howto/`/`notes/`) создавай вместе с её
`README.md`-индексом в момент появления первого файла внутри.

## 6. Чего НЕ делать

- Не дублировать то, что уже есть в **коде, `desc.md`, корневом `README.md`** — ссылайся.
- Не плодить файлы/разделы ради полноты.
- Не хранить **секреты** (ключи API, токены). У climem ключ эмбеддингов и так не живёт в
  репо — он берётся из переменной окружения по имени `embedding.api_key_env` (см.
  [conventions.md](conventions.md)).
- Не оставлять заведомо устаревшее: правишь факт в коде — правь и память.
- Не превращать `journal.md` в пересказ диффа — пиши **решения и почему**.

## 7. Поддержание свежести

- В «живых» файлах ([overview.md](overview.md)) — строка «последняя ревизия: дата».
- Раз в N времени — быстрый проход «что устарело» (особенно версии и активная задача).
- Факт памяти, разошедшийся с кодом, — **чини сразу** (это часть работы, не долг).
- Когда проект встанет под git — идеально добавить Stop-хук, напоминающий обновить память,
  если менялся код, а `memory/` — нет.

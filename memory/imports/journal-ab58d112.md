## 2026-06-30 — `recall` preview-first + `--budget` (заимствование из graphify, экономия токенов)

**Зачем.** Изучали соседний проект graphify (Python, кодовый граф для агентов): его тезис —
построить компактный индекс один раз, потом отвечать из индекса, а не перечитывать файлы
(их бенчмарк «71.5× меньше токенов на запрос»). cm уже на том же тезисе (`map`+`recall`), но
его горячий путь чтения — `recall` — по умолчанию возвращал **полное тело** каждого из 5 хитов.
Самый крупный рычаг экономии: вернуть ограниченный preview, а полное тело — по запросу
(`cm get`). Это ровно graphify RFC «file-level node summaries» + его `query --budget`.

**Что сделано (единый механизм для обоих пунктов).** Per-result бюджет тела в символах
(`output::insert_body`): тело в пределах бюджета → целиком под `body` (короткие `remember`-факты
не меняются, лишних `get`-раундтрипов нет); сверх бюджета → `body` заменяется на `preview`+`chars`
(полная длина — маркер «есть ещё»). Смена ключа `body→preview` намеренна — читающий `body`
замечает отсутствие, а не действует на обрезке.
- Дефолт бюджета: новое поле конфига `search.recall_body_chars = 500` (~80–90 слов; 200-словный
  импортный чанк ужимается примерно втрое, факты проходят целиком).
- `recall --budget N` перебивает на вызов (`commands::parse_budget`; `0`→без обрезки),
  `--full` — тоже без обрезки. С `--fields` бюджет не применяется (явная проекция), warning в stderr.
- `chars` добавлен в `RECALL_FIELDS`/`field_value` (известное/запрашиваемое поле).

**Контракт синхронизирован в одном изменении:** `help::HELP` (recall-секция), `cli::VALUE_FLAGS`
(+`budget`; `--full` булев), `README` (таблица + «Lean recall output»), генерируемый онбординг
(`init.rs`: CM_GUIDE-блок и pointer-блок — длинный матч приходит как `preview`+`chars`).

**Проверка.** 8-арг `recall_value` ловит clippy `too_many_arguments` — узкий `#[allow]` с
причиной (скоры — отдельные f32 у вызывающего, бандл в структуру только ради счётчика — лишняя
церемония). Новые юнит-тесты: `insert_body` (короткое/длинное/без бюджета), `parse_budget`
(дефолт/0/N/мусор). Вживую на скрэтч-сторе: длинная нота → `{chars:1170,preview:"…"}`, короткая
→ `body`; `--full`/`--budget 0` → целиком; `--fields` warning+игнор; `--budget abc` → подсказка.
clippy/fmt чисто, все тесты зелёные.

**Не взяли (и почему):** MCP-сервер graphify (у cm сознательно отрезан), Leiden/community и
semantic-рёбра через LLM (у cm детерминированный оффлайн-гибрид FTS+вектор), SHA256-кэш
инкрементала (экономит CPU, не токены; reindex уже детектит изменения). Экспорт оставили
полноточным (его задача — полный дамп), бюджет только в `recall`.

## 2026-06-25 — `cm doctor`: диагностика целостности + статистика + делегированная починка

**Что.** Новая команда `cm doctor [--fix]` (`doctor.rs`) — проверка здоровья проекта + статистика +
(под `--fix`) безопасная починка. По умолчанию только читает: проверяет раскладку и производный
`store.db` против md-правды и на каждую проблему печатает точную команду-фикс. Проверки: парсится ли
`config.json`; резолвится ли `config.data_dir` (ловит **переименованную** папку — ищет одного
однозначного соседа со `store.db`); есть/открывается ли `store.db` (ловит **удалённый** индекс);
рассинхрон `notes`↔`notes_fts`; дрейф подписи/размерности эмбеддера; дрейф store↔диск (строка БД без
`notes/<id>.md` и наоборот); slug-коллизии; граф кода; **`agent_wiring`** (привязаны ли файлы-инструкции
CLAUDE.md/AGENTS.md/… и указывают ли на СУЩЕСТВУЮЩИЙ бинарь cm в корне — ловит «инструкции ссылаются на
cm, которого нет на диске»); **`note_encoding`** (тела `notes/*.md`/`imports/*` с потерянным не-ASCII —
серии `?`/`U+FFFD`/невалидный UTF-8 — ловит заметки, где кириллицу затёрла оболочка ДО cm; **report-only**,
потеря необратима, фикс = переписать/`forget`; превенция — предупреждение `warn_if_shell_mangled` в
`read_stdin` при сохранении затёртого тела); `scaffold` (info: gitignore/CM_GUIDE). `--fix` чинит
только безопасное, **делегируя**: `commands::reindex`(`--all`) + `mkdir`, и при однозначном
переименовании — перезапись `config.data_dir`. Вывод по умолчанию — JSONL `{check,status,fixable,fix}`
+ `{stat,…}` + итог `{doctor,errors,warnings,info,fixed}`, **всегда exit 0**; флаг **`--text`** даёт
человекочитаемый отчёт (✓/•/⚠/✗ + блок статистики, байты в МБ) для глаз. Реализация: `doctor::run`/`diagnose`/
`store_checks`; 5 новых read-only проб в `store.rs` (`fts_desync_counts`, `resolved_edge_orphans`,
`note_ids_of_kind`, `embedding_stats`, `code_orphans`); `init::layout_status`/`LayoutStatus` +
`is_memory_folder` открыты `pub(crate)` (probe раскладки переиспользуется, не дублируется); регистрация
в `main::dispatch`. Контракт в локстепе: `help::HELP` (навигатор + блок + тест-массив `doctor`),
`README.md` (строка таблицы), комментарий в `cli.rs` (`--fix`/`--text` булевы, `VALUE_FLAGS` не трогали).
Тесты: `doctor::` + `store::` unit, fmt/clippy/тесты зелёные (366 unit), плюс ручной прогон сценариев
healthy → удалили `store.db` → `--fix` (rebuild) → переименовали папку → `--fix` (adopt).

**Почему так.** `init` уже самозалечивает раскладку (`LayoutState`), но не диагностирует ДАННЫЕ и не
помогает при переименованной папке (создал бы пустую `memory/`). doctor закрывает эту нишу как
read-сторона зеркала. Ключевой принцип — **делегировать, а не дублировать**: чтобы probe раскладки был
один, открыли `init::layout_status` вместо копии. `--fix` намеренно узкий (выбор: только индекс/dirs +
переименование), чтобы команда с именем «doctor» не переписывала молча CLAUDE.md/конфиг и не теряла
данные; тяжёлое/неоднозначное — только подсказка. Полная сверка графа кода отложена: `--fix` его всё
равно не чинит (только `cm map`). **Проверка `agent_wiring`** добавлена по фидбеку: реальный баг —
`CLAUDE.md` ссылается на `memory\cm.exe`, которого нет на диске; теперь и doctor, и `init` проверяют,
что файлы-инструкции указывают на существующий бинарь (`init::layout_status` расширен, `init::run`
предупреждает после wiring). Решение и инварианты — [decisions.md](decisions.md) «`cm doctor`».

## 2026-06-25 — `cm feedback`: канал отзыва модели о самом cm в память проекта

**Что.** Новая команда `cm feedback` (+ `cm feedback --list`): агент, использующий cm, пишет, чего
инструменту не хватает / что улучшить, тело через stdin — это сохраняется в память ПРОЕКТА как
обычная заметка с зарезервированным тегом `cm-feedback` и штампом версии в `source`. `--list` читает
их обратно (newest-first). Реализация: `commands::feedback`/`feedback_list`/`merge_feedback_tags`/
`FEEDBACK_TAG` (тонкий сиблинг `remember` без графа), регистрация в `main::dispatch`, новых
value-флагов не потребовалось (`tags`/`source`/`recent` уже есть, `--list` булев). Инструкции вшиты
во все три онбординг-поверхности `init` (`entry_point_block`, `cm_guide`/`CM_GUIDE.md`,
`help::pointer`) + `help::HELP` (навигатор + блок команды + типовая сессия) + README. Тесты:
`merge_feedback_tags_always_includes_reserved_and_dedupes` (unit) + e2e `tests/feedback.rs` (write
→ tagged/versioned md, recall by tag, `--list`, пустой stdin → ошибка). fmt/clippy/тесты зелёные
(351 unit + интеграционные, +3 e2e).

**Почему так.** Модель — настоящий потребитель cm, первой видит шероховатости, но не было способа их
зафиксировать не выходя из CLI. Форма «заметка-с-тегом, а не отдельный стор» выбрана, чтобы отзыв был
md-правдой (переживает потерю `store.db`, идёт через `reindex`/`export`/`deinit` бесплатно, инвариант
(a)/(d)). Отвергнуты: oplog-only (не md-правда, теряется) и отдельная папка (лишняя поверхность). Из
`recall` не исключаем — фильтруется тегом, скрытое исключение противоречило бы контракту `recall`.
Подробно — [decisions.md](decisions.md) «cm feedback».

**Как проверял.** Dogfood релизным бинарём во временном проекте: `init` → `feedback` пишет
`tags: cm-feedback, map` + `source: cm v0.1.0`, `feedback --list` отдаёт обе записи newest-first,
`recall --tag cm-feedback` находит, `CM_GUIDE.md`/`AGENTS.md` несут секцию feedback.

**Состязательное ревью (мульти-агент) поймало 1 реальный баг:** `feedback_list` сортировал по epoch
`created_at`, который `reindex` обнуляет → после любого `reindex` порядок схлопывался в случайный
tie-break по hex-id (не newest-first, молча — вывод показывает `created_iso`, который выглядит верно).
Починено: сортировка по строке `created_iso` (ISO-8601 = хронологический лексикографический порядок,
переживает reindex). Добавлен регресс-тест `feedback_list_is_newest_first_after_reindex` (новее →
меньший id, чтобы регрессия к id-tie-break всплыла). Грабли занесены в [conventions.md](conventions.md).

## 2026-06-23 — `cm map` самозалечивается на пустом запросе (устаревший/непостроенный граф)

**Что.** Любой read-only запрос `cm map` (`--query`/`--like`/`--list`/`--uses`/`--calls`/`--defines`),
вернувший пустой результат, теперь один раз авто-реиндексирует запомненное дерево исходников и
повторяет запрос — прозрачно для модели (нарратив в stderr, stdout остаётся чистым JSONL).
Реализация: `map_tree` пишет абсолютный корень в `meta.code_root` (`CODE_ROOT_META_KEY`) при каждом
`cm map`/`init --code`; query-ветки `commands::map` отрефакторены в `run_code_query` (собирает строки
через `collect_code_query`, на пустоте зовёт `auto_remap_for_query`, повторяет). `display_root` чистит
`\\?\`-префикс от `canonicalize` в нотисах. Обновлены `help::HELP`, `help::pointer`, CLAUDE.md, README,
`memory/` (structure + decisions). Тесты: `run_code_query_auto_remaps_stale_graph`,
`..._absent_symbol_stays_empty_after_one_remap`, `map_tree_records_code_root_in_meta`,
`auto_remap_no_op_when_no_root_recorded`, `display_root_strips_verbatim_prefix` (часть за фичей `code`).

**Почему так.** Две связанные жалобы пользователя: (1) модель редко тянется к `cm map`, чаще grep;
(2) если код не проиндексирован/устарел — `cm map` молча отдаёт **пусто**. (2) — корень (1): пустой
ответ без объяснения читается моделью как «инструмент сломан», и она уходит в grep до конца сессии.
В журнале 2026-06-18 «несвежий кэш → grep» было честной оговоркой в инструкциях модели; теперь это
снято — staleness больше не повод обходить map. Решение делает граф самообновляющимся вместо того,
чтобы учить модель помнить про ручной `cm map` после правок (она не помнит). **Ключевые инварианты:**
реиндекс ≤1 раза за запрос (реально отсутствующий символ → пустота без петли); best-effort (нет
`code_root`/дерево исчезло/нет фичи `code` → пустота остаётся пустотой, не ошибка команды); нарратив
только в stderr; `code_root` абсолютный (работа из любой CWD), в meta ровно один — последний
промапленный корень. Вариант с grep-фолбэком («не нашли по индексу → grep → если grep нашёл →
индексируем») отклонён: cm уже знает корень дерева, поэтому реиндексит напрямую — grep как «нашлось
ли вообще» избыточен.

**Как проверял.** Dogfood на самом репо: правил `src/lib.rs` во временном проекте, `cm map --query <new>`
из чужой CWD → нотис о реиндексе + найденный символ одним вызовом; отсутствующий символ → один
реиндекс, потом честная пустота. На climem: обновил `memory/cm.exe`, `cm map ./src` (766 символов),
новые символы (`run_code_query`/`auto_remap_for_query`/`display_root`) видны, `--query <absent>` ре-мапит
`D:\…\src`. fmt/clippy/тесты зелёные в обеих конфигурациях фичи (362 unit + интеграционные).

## 2026-06-21 — E2E-тест внедрения cm на реальном проекте (iscreen) + фикс потери store.db

**Что.** По запросу прогнал полный жизненный цикл внедрения `cm.exe` на реальном C#-проекте
`d:\programming\my\temp\iscreen` (под git, 57 .cs): deinit старой установки → git-чистка до
baseline → свежий бинарь `--features code` → установка с нуля → повторная установка →
изменение доков → достройка → финальный deinit → восстановление baseline (все хеши сверены,
проект возвращён 1-в-1). По ходу — найден и исправлен ДЕФЕКТ.

**Дефект (найден на шаге достройки).** После удаления производного `store.db` повторный `init`
(repair) пересоздавал ПУСТОЙ store и переиндексировал только КОД (шаг [4/N] `map`), а документы
из `imports/` НЕ переиндексировал → `recall` молчал до ручного `cm reindex`. Противоречит
ожиданию «store.db производный → init его восстанавливает». **Фикс:** в `init::run` после
скаффолда, если `!pre.store && has_md_truth(data_folder)` (store создавался заново И есть ≥1 .md
в notes/ или imports/), вызываем `reindex_documents` = `commands::reindex(&Parsed::parse(&[]),
&Ctx::new(target))` (best-effort, не валит init; печатает своё `{"indexed","changed"}`). Гейт
точечный: свежая установка (пустые папки) и present-store не триггерят. Новые хелперы
`has_md_truth`/`reindex_documents` (init.rs). Код-граф по-прежнему отдельно в шаге [4/N].

**Прочие наблюдения (поведение корректное, отметил).** (1) deinit восстанавливает импортированные
доки в `docs/climem/` если оригиналы на месте (конфликт-дедуп, без потерь). (2) `gitignore:
untouched` если между init-ами потерян манифест (мой тестовый `rm -rf memory`) — точность отката
.gitignore зависит от манифеста. (3) повторный `--docs` тех же доков дублирует (import не
идемпотентен — известно).

**Как проверял.** Живьём в /tmp: install→del store.db→`init`→`recall` находит док СРАЗУ
(stderr «store.db was missing — rebuilding the document index from md…»); fresh install и
present-store reindex НЕ запускают. Юнит-тесты: `init_rebuilds_document_index_after_store_loss`,
`init_fresh_install_does_not_reindex_documents`. Гейт: 397 тестов (default) / 406 (code), clippy
(обе фичи) и fmt чисто. Контракт: `help::HELP` (IDEMPOTENT), `README.md`. Память: правка note
[[init-idempotent-repair]].

## 2026-06-21 — Два крайних случая идемпотентного `init` (после тестирования init/deinit)

**Что.** Живое тестирование `init`/`deinit` (данные есть / из корня / повторно, сборка с
`--features code`) выявило два пограничных случая по СЕМАНТИКЕ — данные раскладывались верно,
расходились ярлыки/поведение. Исправил оба:

1. **`init --docs <paths>` на ПОЛНОЙ раскладке игнорировал импорт.** `is_complete` срабатывал
   раньше шага docs → `already_exists`, доки не подхватывались. Стало: short-circuit
   пропускается, если передан явный непустой `--docs` (`explicit_docs =
   p.value("docs").is_some_and(docs_arg_has_entry)` — тот же фильтр пустоты, что у шага
   импорта, так что `--docs ""`/`--docs ,` всё ещё допускают no-op). Тело отрабатывает, доки
   импортируются в существующий store; бара `init` без `--docs` на полной раскладке по-прежнему
   no-op. Подсказка в `already_exists` обновлена: «pass --docs <paths> to import more».

2. **`init` после `deinit` репортил `repaired`, а не `created`.** `deinit` намеренно ОСТАВЛЯЕТ
   `config.json` в корне, а `any_present()` считал config признаком достройки. Убрал `config`
   из `any_present()` (теперь только store/dirs/data-.gitignore/guide = «данные прошлого init
   уже здесь»). Одинокий config — after-state деинита → init поверх него = свежий `created`
   (config при этом всё равно переиспользуется, не перезаписывается).

**Почему.** Оба — про ожидания пользователя: «доложить доки в готовую память» (теперь работает
через `init --docs`, не только `cm import`), и «после полного uninstall+reinstall это свежая
установка». Поведение по данным не менялось — менялись триггер short-circuit и ярлык status.

**Как проверял.** `cargo test --release --features code` — 355 unit зелёные (новые
`init_with_docs_imports_into_complete_layout`, расширен `layout_state_probe_and_completeness`
кейсом lone-config); `--features` по умолчанию (api, без code) — 395 суммарно; `clippy
--all-targets` (и с `code`, и без) чисто; `fmt` чисто. Живьём в /tmp: `init --docs extra` на
полной раскладке → `repaired`, `imports/late.md` создан, recall находит chunk; `init`→`deinit`
→`init` → третий запуск `created` (был бы `repaired`). Контракт: `help::HELP` (секция
IDEMPOTENT), `README.md` (врезка). Память: правка note [[init-idempotent-repair]].

## 2026-06-20 — `init` стал идемпотентным: достраивает раскладку, не упирается в «папка есть»

**Что.** По запросу пользователя («нужна более умная проверка и до-инициализация по init»)
заменил бинарный guard `init::run`: было `if data_folder.exists() { already_exists; return }`
— любая папка с именем `memory/` (даже пустая) блокировала init. Стало: `init` приводит
проект к ПОЛНОСТЬЮ инициализированному состоянию и трогает только недостающее.
Реализация: новый `struct LayoutState` + `probe(target, data_folder, config_path)` (чистые
ФС-чтения до правок) снимает наличие каждого артефакта: config.json, store.db,
notes/imports/models, data-`.gitignore`, бинарь в корне, покрытие cm.exe в root-`.gitignore`,
CM_GUIDE.md. `is_complete(target, exe_display)` = все части И `instruction_pointer_current`
(хотя бы один instruction-файл несёт БАЙТ-идентичный текущему pointer-блок — устаревший путь

## 2026-06-20 — Прогресс-бар при индексации исходников в `init` (шаг [4/4])

**Что.** На шаге `[4/4] Indexing source code…` `init` молча висел со спиннером, пока
`map_tree` парсил всё дерево (сотни файлов) — выглядело как зависание. Добавил живой
ASCII-прогресс-бар на месте: `      [######------------]  47%  142/300`, перерисовывается
на той же строке через `\r`. `commands::map_tree` получил новый опциональный параметр
`progress: Option<&dyn Fn(usize, usize)>`, вызываемый раз на каждый обойдённый файл
`(i+1, total)` ДО его парса (так бар доходит до `total/total` ровно). `cm map` передаёт
`None` (его вывод — JSONL, бар не нужен); `init::map_source_tree` передаёт `progress_bar`.
Верстку вынес в чистую `init::render_bar(done,total) -> String` (юнит-тесты на точную
строку: старт/середина/конец + clamp при `done>total`); `progress_bar` лишь оборачивает её
в `\r…\x1b[K` и флашит. По завершении/при ошибке строка стирается `clear_progress_line`.

**Почему так.** (1) Бар включается **только когда stderr — терминал** (`std::io::IsTerminal`,
stdlib, без новых зависимостей — инвариант минимума deps): при редиректе в файл/пайп (CI, лог)
`\r`-анимация замусорила бы лог, поэтому там тихо — остаётся лишь финальная строка-сводка
`N symbol(s) in M file(s)`. Подтверждено побайтово: в редиректнутый stderr не утекает ни
`\x1b`, ни сигнатур бара (`[#`, `%␠␠`). (2) Колбэк — чистый display-хук: индексация
идентична с ним и без; `None` индексирует ровно так же, просто без бара. (3) JSONL на stdout
не тронут (conventions.md: нарратив — только stderr). Поведение `cm map` байт-в-байт прежнее.

**Как проверял.** fmt/clippy чисто (с фичей `code` и без), `cargo test --features code` зелёный
(351 unit + интеграционные), новые `render_bar_layout_endpoints_and_midpoint` /
`render_bar_clamps_overrun` проходят. Dogfood `cm init` на скретч-дереве: в не-TTY бар подавлен,
сводка `5 symbol(s) in 5 file(s) [Rust 5]` печатается; редиректнутый stderr без CR-баров/ESC.
Связано: [[init-step-output]].

## 2026-06-18 — `cm map` Tier 1: --like / --list / --calls + --kind (меньше grep)

**Что.** Четыре новых режима запроса к графу кода (всё read-only, данные уже были в БД):
`--like <substr>` (символы по подстроке имени, `code_symbols_like`), `--list` (оглавление всех
символов, `code_list`), `--calls <name>` (исходящие зависимости символа, `code_callees_of`),
плюс общий фильтр `--kind <function|method|class|…>` для листингов (`commands::filter_kind`).
Флаги в `cli::VALUE_FLAGS` (`like`/`calls`/`kind`; `list`/`external` — булевы). Контракт
(`help::HELP` picker+блок `map`), README, тесты (`store::tests::code_symbols_like/_callees_of/
_list`).

**Почему так.** После тестирования «помогает ли реально искать в коде» (см. предыдущую запись /
ответ пользователю) выявил три дыры, где приходилось возвращаться к grep: поиск по подстроке
имени, «от чего зависит символ», оглавление модуля. Все три закрываются дёшево — нужные рёбра
(`uses`) и узлы уже в графе, не хватало SELECT'ов и флагов. **Ключевое решение в `--calls`:
resolved-only по умолчанию.** Из ~4200 `uses`-рёбер только ~1100 разрешены внутрипроектно;
остальные ~3000 — stdlib/комбинаторы (`map`/`unwrap`/`collect`/…), которые без фильтра топят
реальные зависимости. Поэтому `--calls` по умолчанию показывает только `dst IS NOT NULL`, а
`--external` — всё. Без этого фильтра фича была бы мертворождённой.

**Граница (честно, НЕ баг).** `--uses`/`--calls` резолвятся по имени без разрешения областей:
одноимённые символы сливаются, возможен ложный хит (напр. в `--calls reindex_notes` проскочили
проектные `map`/`all` как «resolved»). Задокументировано в help. Точный резолв — это stack-graphs
(Tier 3, отдельная задача). Полнотекст по телам/комментам и индексацию `.md`/`.json` НЕ делаем —
для текста есть grep и `import`/`recall`; граф остаётся про структуру. **grep полностью не
заменяется и это не цель** — Tier 1 лишь убирает частые *структурные* обращения к нему.

**Как проверял.** Dogfood на src climem: `--like code` → 48 символов `code_*`; `--calls
reindex_notes` → 15 внутрипроектных зависимостей (vs 48 с `--external`); `--list --kind class`

## 2026-06-18 — `init --code`: индексировать исходники сразу при init

**Что.** Флаг `cm init <path> --code`: после разворачивания папки памяти `init` индексирует
дерево исходников `<path>` в граф кода. Замыкает исходный UX-сценарий («положил exe в корень
проекта → `cm init . --code` → готовая карта кода в один шаг»). Вынес ядро индексации
`commands::map_tree` (`MapStats`) из `commands::map` — теперь его зовут и `cm map`, и
`init::map_source_tree`. Счётчики кода добавляются в JSON init под ключом `code`. Контракт
(`help::HELP`), README, `memory/` обновлены; тесты на обе конфигурации фичи.

**Почему так.** В предыдущей сессии `init --code` был сознательно отложен (чтобы не смешивать
поведения, пока `cm map` не докажет пользу) — теперь доделан по запросу. **Best-effort —
ключевое решение:** скаффолд (store/config/notes/imports) уже создан к моменту индексации,
поэтому код-маппинг **не имеет права уронить `init`**: без фичи `code` (или при ошибке парса)
печатаем warning+hint в stderr и продолжаем, ровно как bulk-импорт md в том же `init`. Папка
памяти передаётся в `map_tree` как `mem_dir` → не индексирует собственный бинарь/store. По
умолчанию (`init` без `--code`) поведение байт-в-байт прежнее.

**Как проверял.** Dogfood: `cm init . --name mem --code` на дереве Python+Go → JSON с
`"code":{files,symbols,edges}`, `cm map --query` находит символы; папка памяти не попала в
граф. Без фичи: warning+hint, но `store.db` создан (скаффолд цел). Юнит-тесты:
`init_code_indexes_source_tree` (под фичей), `init_code_still_scaffolds_when_nothing_to_index`
(обе конфигурации). fmt/clippy чисто, тесты зелёные.

## 2026-06-18 — Граф кода (`cm map`, tree-sitter), ОТДЕЛЬНЫЙ от графа заметок

**Что.** Новая подсистема: отдельный граф знаний для **исходного кода** за фичей `code`.
Модуль [code.rs](code.rs) (реестр 11 языков + извлечение defines/uses через `tree-sitter-tags`
по `tags.scm` грамматики), таблицы `code_files`/`code_symbols`/`code_edges` в [store.rs](store.rs)
(+ методы), команда `cm map` в [commands.rs](commands.rs) (диспетч в `main.rs`, флаги в `cli.rs`).
Индексация: `cm map <path>` (рекурсия, скип build/vendor, инкремент по content_hash); запросы
`--query <name>` (где определён), `--uses <name>` (кто использует, с охватывающим символом),
`--defines <file>` (что определяет файл, путь матчится нестрого). Языки: Rust/Python/JS/TS/Go/
Java/C/C++/C#/Ruby/PHP. Контракт в `help::HELP` + README обновлены.

**Почему так.** Пользователь хотел дать модели карту исходников. Ключевое: в climem **уже был**
движок графа (таблица `edges`, `related`/`backlinks`, висячие рёбра) — не хватало лишь
*источника рёбер из кода*. По явному требованию пользователя граф кода сделан **ОТДЕЛЬНЫМ** от
графа заметок (свои таблицы, своя команда, не лезет в `recall`/`related`): код и заметки —
разные сущности, смешав, утопили бы «почему»-заметки в кусках кода. Не свой парсер, а
tree-sitter + `tree-sitter-tags` (механизм go-to-definition GitHub'а): единый поток тегов на
все языки → один код извлечения, добавить язык = крейт + строка в `LANGUAGES`. Полное
обоснование и инварианты — [decisions.md](decisions.md) «Граф кода».

**Как проверял.** ABI-проба (ядро ts 0.26 + грамматики 0.23–0.25 совместимы через
`tree-sitter-language`; `tags.scm` отдаёт def/ref на rust/python/go) — до написания подсистемы.
Dogfood на самом climem: `cm map ./src` → 20 файлов, 631 символ, 4763 ребра; `--query
upsert_note`/`map`, `--uses upsert_note` (сверено с grep — все вызовы найдены и привязаны к
охватывающей функции), `--defines` во всех трёх формах пути. Инкремент стабилен (`changed:0` без
правок). Сборка с/без фичи, `--features code` тесты (`parse_rust`/`parse_python`) зелёные.

**Остаётся (сознательно НЕ сделано).** Граф вызовов с разрешением областей (stack-graphs) — `uses`
сейчас эвристика по имени. Крючок `init --code` (авто-индекс при init) — отложен, чтобы не
смешивать поведения. Многоязычность можно расширять реестром без миграции схемы (есть колонка
`lang`). **NB:** `memory/` — активный cm-store; эти .md правились как `imports/`-копии (баг
«import not idempotent» — отдельно).

## 2026-06-18 — Команда `deinit`: снять производные следы cm, оставив md-правду

**Что.** Новая команда `cm deinit <target> [--name <name>] [--yes]` — зеркало `init`. Модуль
[deinit.rs](deinit.rs), диспетч в `main.rs` рядом с `init` (тоже **не** открывает store —
работает с ФС напрямую). Удаляет только **производное**: `store.db`(+`-wal`/`-shm`),
`config.json`, `.gitignore` (тот, что писал init), `models/`, и pointer-блоки из файлов-
инструкций (через переиспользуемый `init::unwire_entry_points`). **Сохраняет** источник правды —
`notes/*.md` + `imports/*` — и копию `cm(.exe)` (её юзер удаляет последней; запущенный exe на
Windows сам себя не снесёт). Если папка памяти после этого пуста — сносится целиком
(`remove_if_empty` сначала прунит пустые `notes/`/`imports/`). Подтверждение y/N (переиспользует
`init::prompt_yes_no`), `--yes`/`--force` пропускает, piped stdin (EOF) безопасно отказывает.
Вывод JSONL: `{"deinit","removed":[…],"unwired_files":N,"folder_removed":bool}`.

**Wiring стал двунаправленным.** Чтобы `deinit` снимал ровно то, что `init` дописал, добавлен
`init::strip_block` (обратное к `replace_block`): вырезает marker-блок И тот единственный
blank-line-разделитель, что вставлял `wire_entry_points`, так что once-wired файл
**round-trip'ится в исходные байты** (файл без финального `\n` получает один — мы всегда
терминируем восстановленный контент). Константы/хелперы wiring (`ENTRY_POINT_NAMES`,
`WIRE_BEGIN`/`WIRE_END`, `prompt_yes_no`, `unwire_entry_points`) подняты до `pub(crate)`.

**Почему важно (по запросу пользователя).** Нужен был чистый «откат»: убрать следы cm так, чтобы
осталось только удалить exe, **не теряя пользовательский контент**. Поэтому граница — ровно
truth↔derived: всё в `DERIVED_FILES`/`DERIVED_DIRS` пересобираемо (`cm reindex` из md), а
`notes/`/`imports/` — нет. Худшая цена ошибочного `--yes` — один `reindex`, не потеря памяти.
exe не трогаем сознательно (Windows-ограничение + явная формулировка пользователя «останется
только удалить exe»).

**Как проверял.** `cargo test --release` — 286 unit/integration + e2e зелёные (6 тестов в
`deinit.rs`: keeps-truth/removes-derived, прун пустой папки, отказ без `--yes` на piped stdin,
hint при отсутствии target, `--force`-алиас, кастомный `--name`; + в `init.rs`
`wire_then_unwire_roundtrips_to_original`, `strip_block_*`, `unwire_*`). `clippy --all-targets` +
`fmt --check` чисто. **Вживую** в `/tmp/cmdeinit`: init→remember→deinit — note `notes/<id>.md` и
`cm.exe` уцелели, `store.db`/`config.json`/`.gitignore`/`models/` удалены, CLAUDE.md вернулся к
`project rules\n` (0 блоков), пустой `imports/` свёрнут.

**Что осталось.** `deinit` находит точки входа по `<target>` (как `init`), не по `resolve_dir` —
значит для нестандартной раскладки надо передать тот же путь, что и в `init` (задокументировано в
help/README). Контракт синхронизирован: `help::HELP` (новый блок + навигатор + тест
`help_const_mentions_all_commands`), README (строка таблицы + source layout),
[structure.md](structure.md).

## 2026-06-18 — `init` авто-привязывает файлы-инструкции агента к store

**Что.** `init` после bulk-импорта `.md` теперь ищет в target файлы-«точки входа» агента
(`ENTRY_POINT_NAMES`: CLAUDE.md, AGENTS.md, AGENT.md, GEMINI.md, .cursorrules,
.github/copilot-instructions.md) и **дописывает** в каждый найденный pointer-блок: «доки проекта
теперь бери через `cm recall`, а не читай напрямую; после решений — `cm remember`». Реализация —
`init::wire_entry_points`/`entry_point_block`, блок обрамлён маркерами `WIRE_BEGIN`/`WIRE_END`
(`<!-- BEGIN/END cm memory pointer -->`). Файлы **не создаются**, только правятся существующие;
ошибки I/O — нефатальный warning в stderr, успех — `{"wired": "<path>"}` в stdout (JSONL-инвариант
через `output::print_line`). Контракт синхронизирован: `help::HELP` (блок `init`), README (врезка
«Auto-wiring agent instructions»), [structure.md](structure.md) (строка `init.rs`).

**Идемпотентность с самоисцелением (по вопросу пользователя «что при повторном init?»).**
Разобрали два сценария повтора: (1) **то же `--name`** — guard `folder.exists()` в `init::run`
делает ранний `already_exists`-return, wiring вообще не запускается; (2) **другое `--name`** —
guard пропускает (новая папка), тело выполняется заново, в т.ч. wiring. Поэтому одной «проверки
наличия маркера» мало: при сценарии (2) в CLAUDE.md остался бы указатель на путь ПЕРВОЙ папки.
Решение (выбор пользователя — «обновлять только если путь изменился»): `replace_block` вырезает
старый блок между маркерами и сравнивает с новым — идентичный → `None` (молчаливый no-op, без
лишнего `{"wired"}`), отличается → замена на месте (устаревший exe-путь самоисцеляется), маркера
нет → дописать в конец. Malformed-регион (BEGIN без END) не трогаем.

**Почему важно.** Указатель `help::pointer` печатался в stderr — его надо было вручную копировать
в CLAUDE.md. Теперь `init` сам встраивает инструкцию в те файлы, которые агент и так читает в
начале сессии, замыкая цикл «развернул память → агент знает, что доки лежат в `cm`, а не в md».
Маркерный блок (а не перезапись/импорт самого файла) выбран осознанно: исходные правила
сохраняются, правка обратима, распознаётся по маркеру и переживает re-init под новым путём.

**Как проверял.** `cargo test --release` — 277 unit/integration + e2e зелёные (тесты
`wire_entry_points_*`, `wire_entry_points_refreshes_stale_path_in_place`, `replace_block_*`,
`init_wires_existing_claude_md`); `clippy --all-targets` чисто. Плюс **вживую через `cm.exe`**: три
точки входа (вкл. вложенный `.github/copilot-instructions.md`) получили блок, README.md не тронут,
повторный init с тем же именем — no-op (`already_exists`), с другим именем (`.memory2`) — путь в
CLAUDE.md обновился на новый, блок остался ровно один.

**Попутно.** Удалён мёртвый e2e `tests/cli_help.rs::mcp_exits_2` — команда-заглушка `mcp` была
вырезана из `main.rs` в этой же сессии (см. [roadmap.md](roadmap.md)/[structure.md](structure.md)),
а тест продолжал ждать exit 2; путь «неизвестная команда → exit 1» уже покрыт
`unknown_command_exits_1_with_hint`.

**Что осталось.** Имя файла матчится по точному регистру `ENTRY_POINT_NAMES` (не
case-insensitive по имени) — на NTFS это неважно, на Linux `claude.md` не подхватится; оставлено

## 2026-06-18 — bulk-импорт `init` стал рекурсивным

**Что сделано.** `collect_md_files` в [../src/init.rs](../src/init.rs) теперь обходит target
**рекурсивно** (ручной DFS `walk_md`, без `walkdir` — инвариант минимума зависимостей), а не
только верхний уровень. Сигнатура сменилась на `collect_md_files(dir, exclude)`: второй аргумент —
только что созданная папка памяти (`folder`), её обход пропускается целиком, чтобы копии в
`imports/` не засосало обратно при `init ./` (когда `folder` лежит внутри `target`). Хелпер
`is_md` вынесен из замыкания (используется в обоих местах). Симлинк-директории не обходятся:
`entry.file_type()` из `read_dir` на всех ОС отражает сам линк, поэтому `is_dir()` для него ложен —
заодно защита от циклов/выхода за дерево доков. Промпт обновлён: «(включая вложенные папки)».

**Почему.** Типовой сценарий «папка с доками» (см. комментарий в `import_existing_md`) на деле
часто иерархичен: `docs/guide/intro.md`, `docs/api/*.md`. Старое поведение (только верхний
уровень) молча пропускало вложенное, и пользователь думал, что всё импортировано. Рекурсия
закрывает дыру; исключение папки памяти — чтобы рекурсия не создала петлю самопоглощения.

**Баг, пойманный живым тестом.** Первая версия исключала из обхода только *свою* новую папку
(`exclude`). Ручной прогон `init` второй раз в той же папке (другое имя) показал «Найдено 8»
вместо 4: рекурсия засосала `imports/`-копии *ранее созданной* папки памяти, для которой
`exclude` не совпадал. Это петля самопоглощения между двумя памятями. Починка: `walk_md` теперь

## 2026-06-18 — интерактивный импорт при `init` ("docs folder" режим)

**Что сделано.** `init` теперь определяет, есть ли `.md` файлы в target-директории, и если да —
предлагает их импортировать интерактивно. Логика в `src/init.rs`:
- `collect_md_files(target)` — сканирует target (не рекурсивно), возвращает отсортированный
  список `.md`-файлов.
- `prompt_yes_no(question)` — печатает вопрос в stderr, читает строку из stdin; `true` для
  `y/yes/д/да`; `false` при EOF/ошибке (пайп → тихий пропуск, не ломает скрипты).
- `import_existing_md(...)` — если пользователь ответил `y`: строит эмбеддер, импортирует
  каждый файл через `import::import_file`, выводит JSONL на stdout. Потом спрашивает про
  удаление оригиналов (только если хоть один успешно импортирован).

**Почему.** Сценарий: пользователь кладёт `memory.exe` в папку с документами, запускает
`memory init ./` — прога видит `.md` файлы и предлагает их всосать сразу. Без этого пришлось бы
вызывать `import` по одному.

**Инварианты сохранены.** stdin для интерактивного ввода — только в `init` (не конфликтует с
`read_stdin` в `remember`, который читает тело заметки из stdin в других командах). Импорт идёт
через стандартный `import::import_file` — идемпотентен, копирует оригинал в `imports/` до
индексации. JSONL на stdout, нарратив в stderr.

**Само-ревью + ручное тестирование (5 сценариев Bash + PowerShell).** Нашёл и починил два бага:
- **BOM в пайпе под PowerShell.** `"y" | memory init` НЕ срабатывал — PS добавляет BOM, и
  `"\u{feff}y" != "y"`. Та же грабля, что чинит `read_stdin` (conventions.md). Вынес парсинг в
  чистую `is_yes()`, которая стрипит BOM перед сравнением. Интерактивный ввод (печать y+Enter)
  BOM не имеет — это артефакт редиректа, так что основной путь и так работал; фикс — для пайпа.
- **Case-sensitive расширение.** `collect_md_files` матчил только строчное `.md`, а `import`
  принимает `.md`/`.markdown` case-insensitively → `README.MD` молча пропускался. Выровнял:
  скан теперь lowercase'ит расширение и принимает оба, совпадая с тем, что реально умеет импорт.

Добавлены юнит-тесты на чистые хелперы (`is_yes`, `collect_md_files`); README дополнен описанием
режима «папка с доками». Все 255 unit + интеграционные тесты зелёные, clippy чист.

## 2026-06-17 — тестирование CLI + фикс BOM в read_stdin (Windows PowerShell)

**Что сделано.** Прогнал полный smoke-тест `memory.exe`: `init`, `remember` (stdin), `recall`

## 2026-06-17 — смена концепции хранения: md = правда, store.db = производный индекс + граф знаний

**Что сделано.** Реализована задумка [`desc.md §3`](../desc.md) (она существовала только на бумаге —
код хранил всё в `store.db` как в единственном источнике правды). Теперь **источник правды — md**:
`notes/<id>.md` (1 файл = 1 заметка, ручной `---`-frontmatter + тело) и `imports/` (оригиналы +
`<name>.meta.json`-сайдкары с orig-именем и тегами). `store.db` — **производный, пересобираемый**
индекс (FTS5 + векторы + граф). По фазам:
- **id → hex-строка.** Атомарный флип `i64 → String` по всему коду (`NoteRow.id`, get/forget/list/
  all/ids_matching/all_embeddings/fts_search, output, ранжирование в `search`, `parse_id` теперь
  валидирует `^[0-9a-f]+$`). id минтится (`store::fresh_id`/`mint_hex`, FNV-фолд time+pid+counter,
  collision-check), `id == notes/<id>.md`. FTS5 держит внутренний INTEGER rowid; `fts_search`
  джойнит `notes` по rowid и отдаёт публичный hex; `upsert_note` — явный DELETE+INSERT (rowid не виснет).
- **note.rs** — ручной парсер/рендер frontmatter (без serde_yaml), байт-стабильный фикс-порядок ключей.
- **remember** пишет `notes/<id>.md` (**точка коммита**), затем индексирует best-effort (сбой
  эмбеддера → предупреждение в stderr, заметка цела, `{"id"}` печатается), пишет sync, выводит рёбра
  из `[[вики-ссылок]]` тела. **remember больше не падает на сломанном эмбеддере.**
- **forget** удаляет md + строку + рёбра; **kind-aware**: чанк импорта удалить нельзя (подсказка).
- **import** копирует оригинал в `imports/` + сайдкар (truth) ДО индексации; чанки — производные,
  **контент-адресные** id `content_hash(file_hash#n)`, `source` = копия в imports/. Идемпотентно.
- **reindex [--all]** — пересборка из notes/+imports/. Инкрементально по `content_hash` (таблица
  `sync`); `--all` → `wipe_derived` (стирает notes/fts/sync/imports/edges, НЕ oplog/meta) + переэмбед.
  Двухпроходно (сбор slug→id, затем рёбра — forward-ссылки сходятся). Прунинг исчезнувших файлов.
  **Legacy-guard**: `--all` отказывается стирать, если в БД есть заметки, а `notes/` пуст. Деградирует
  до FTS-only без эмбеддера.
- **граф знаний** (`graph.rs` + таблица `edges`): рёбра из frontmatter `relations` и `[[вики-ссылок]]`
  (`links_to`). `related <id>` — детерминированный BFS, «худой» JSONL, висячие цели первым классом.
- **init** разворачивает notes/+imports/ + пишет `.gitignore` в папку памяти (игнорит производную
  store.db). `config.version` 1 → 2.

**Почему так (ключевые решения).**
- **id = случайный hex во frontmatter, а не autoincrement.** БД производна — счётчик БД не может быть
  идентичностью (ребилд перенумеровал бы и сломал get/forget/связи). desc.md §3 буквально показывает
  `id: 0a1b2c`. Цена — широкий ripple i64→String, принят как атомарный коммит (тип-система неделима).
- **md — точка коммита, индекс — best-effort.** desc.md §3/§10: запись файла — коммит, индекс
  пересобираем. Поэтому remember пишет md ДО сборки эмбеддера и не падает на его сбое (раньше падал).
- **imports/ самоописателен (сайдкары).** Теги импорта не выводятся из файла; держим их в
  `imports/<name>.meta.json` (truth), а не только в БД — тогда **удаление store.db не теряет теги**,
  reindex восстанавливает и контент, и метаданные (иначе — дыра в §10).
- **Разрешение цели графа — slug по умолчанию, префикс `id:` форсирует id.** Никакой эвристики «похоже
  на hex» (8-hex slug мог бы перехватить адресацию) — убрал целый класс ложных срабатываний.
  Висячее ребро — первоклассно (dst_id NULL, dst_raw хранится), оживает на reindex.
- **edges/imports — производные, выводятся из md; oplog/meta переживают wipe** (их из файлов не
  собрать). Всё, что знает индекс/граф, выразимо в md+сайдкарах — иначе БД перестаёт быть одноразовой.
- **Граф — отдельный глагол `related`, не в `recall`.** Сохранил неизменным нагруженный путь
  ранжирования RRF (детерминизм/`score==fts+vector`); RRF-канал графа спроектирован, но отложен.

**Процесс.** Перед кодом — два состязательных воркфлоу (анализ влияния миграции; дизайн графа) с
критиком: нашли блокер (legacy-данные при `--all`), идентичность/forget чанков, не-атомарность
флипа id, FTS-rowid↔hex — все учтены. Сборка велась фазами с зелёными тестами между ними.

**Состояние.** `cargo test` зелено: **249 in-module + 45 e2e** (default / `--no-default-features` /
`--all-features`); `clippy --all-targets -D warnings` чисто; `fmt` чисто; релиз + `--features pdf`
собираются. `help::HELP`, `README.md`, `desc.md §5/§6` синхронны контракту.

`recall --related <id>`** — третий RRF-канал (`score = fts+vector+graph`, `search::recall_with` +
`neighbor_ranks`), вес `search.hybrid_weights.graph` по умолчанию 0.0 (канал выкл → plain `recall`
байт-в-байт прежний); (2) **флаги `remember --slug`/`--relations`** (`parse_relations`, во frontmatter

## 2026-06-18 — реализован план токен-эффективности, Фазы 0–3 ([docs/token-efficiency-plan.md](../docs/token-efficiency-plan.md))

**Что сделано.** Второй порядок экономии токенов на `recall` (главный выигрыш — retrieval — уже
был). По фазам плана:
- **Фаза 0 — мерило.** [scripts/bench.py](../scripts/bench.py): импортирует `memory/`, гоняет
  канонические запросы, считает токены (`tiktoken cl100k_base`) lean vs `--explain` и проверяет
  **recall@k-страж** (ожидаемый `origin` в top-k). На текущей KB: recall@5 = 8/8, lean ≈ на
  7.6 % меньше `--explain` (это только дельта тройки чисел; основной E1/C1-выигрыш — против
  старого формата). Не часть `cargo test`.
- **Фаза 1 (E1, C1, A2).** `recall` по умолчанию печатает «худую» строку (`output::recall_value`):
  `id, kind, body` + непустые `tags`/`origin`/`source`; `score/fts/vector` — под `--explain`;
  `--fields a,b,c` — whitelist `RECALL_FIELDS`. Дефолт чанка 512/64 → **200/32 слов** (`max_tokens`
  переосмыслён как слова, имя сохранено ради совместимости). Ключи JSON — алфавитно/байт-стабильно.
- **Фаза 2 (R1–R4).** RRF вместо min-max (`search::recall_with`, `score = Σ w/(k+rank)`,
  `k=search.rrf_k=60`); глубокий лимит-независимый пул FTS (`search.candidates=50`); полный
  breadcrumb заголовков в `chunk::markdown` (`origin` = `файл › H1 › H2 › H3` + префикс предков
  в тело, бюджет в словах); pre-filter `--tag`/`--origin-prefix` (`store::ids_matching`).
- **Фаза 3 (K1).** `--min-score` (порог в RRF-шкале) + дефолт `recall --limit` 8 → **5**.

**Почему так (ключевые решения).**
- **RRF, не нормированная min-max** (решение — [notes/decisions.md](notes/decisions.md)). Сливает
  *позиции*, а не несопоставимые bm25/cosine → стабильный top-k под безопасную k-резку. Цена:
  отчётные `score/fts/vector` стали мелкими (`~w/(k+rank)`, не [0,1]); `score == fts+vector`.
  Заодно унифицировал кандидатов на `BTreeSet` + tie-break по `id` → **закрыл баг tie-order**
  ([known-issues.md](known-issues.md)).
- **Сигнатуру `search::recall` не ломал**: добавил `recall_with(.., &RecallOpts)`, а `recall(.., limit)`
  оставил тонкой обёрткой — `export` и ~15 unit-тестов не тронуты.
- **Lean by default, полнота по запросу.** Контракт остался JSONL; полная запись — `get <id>`;
  отладка — `--explain`. Срезает «конверт» без потери качества выдачи.
- **`max_tokens` не переименовывал** — сломало бы существующие конфиги; задокументировал как
  «слова» в коде и [conventions.md](conventions.md), снизил дефолт.
- **Фаза 4 (X1–X5) не делал** — план сам помечает её opt-in/по росту KB; X4/X5 вдобавок
  `api`-only. Зафиксирована в [roadmap.md](roadmap.md).

**Отклонения/находки.** `round4` вынесен в `output.rs` (используется проекцией) — возвращает
*расширенный f32* f64, поэтому в юнит-тестах сравнение с допуском, не точное JSON-равенство.
Удалены `norm01`/`min_max*` и их тесты (RRF их не использует). Два chunk-теста и два search-теста
переписаны под новое поведение. `cargo test` (default + `--no-default-features`), `clippy
--all-targets --all-features -D warnings`, `cargo fmt` — зелено.

**Что осталось.** Фаза 4 (preview-then-get, extractive-эксцерпт, дедуп при импорте; reranker/
RAPTOR за `api`). Два regression-anchor'а в [known-issues.md](known-issues.md) (утечка маскирования,
частичный импорт) — по-прежнему открыты, вне этой задачи.

## 2026-06-18 — Фикс: C# (.cs) не индексировался — невалидный `@module` в tags.scm

**Что.** На реальном проекте (iscreen, Avalonia/C#) все `.cs` падали с
`tags query for 'csharp' is invalid: Invalid capture @module. Expected one of:
@definition.*, @reference.*, @doc, @name, @local.(scope|definition|reference)`.
Причина: `tree-sitter-c-sharp` 0.23 в `tags.scm` имеет лишнюю стансу
`(namespace_declaration name: (identifier) @name) @module` — голый `@module`
(дубль валидной строки `@definition.module` прямо над ней). `tree-sitter-tags` 0.26
отвергает ВЕСЬ query при любом неизвестном capture → C# не индексировался совсем
(0 символов на 57 файлов).

Фикс — `code::sanitize_tags_query`: режет стансу (стансы разделены пустой строкой во
всех шипнутых tags.scm), если её КЛАССИФИЦИРУЮЩИЙ (последний `@token`) не из
`ALLOWED_TAG_CAPTURES` (`@definition.`/`@reference.`/`@local.`/`@name`/`@doc`/`@ignore`).
Внутренние `@name` не трогаются. Применяется централизованно в `parse()` ко всем
грамматикам (защита от того же класса бага у других языков), не только к C#. Namespace
по-прежнему тегается через валидную строку выше — потерь нет.

**Также важно (вопрос пользователя «почему удаляются исходники»).** Разобрали: исходники
(.cs и любой код) НЕ удаляются — графовый индексатор их только читает. Промпт «Удалить
исходные файлы? y» в init удаляет ТОЛЬКО успешно импортированные `.md` (их копии лежат в
`imports/`, восстановимы). `imported_paths` (init.rs) наполняется исключительно из
`md_files`; .cs туда не попадают. На iscreen восстановили docs/*.md из imports/.

Проверка: новые тесты `parse_csharp_handles_bad_module_capture_in_tags_query`,
`sanitize_tags_query_drops_only_disallowed_stanzas`. Прогон по rb/php/go/java — символы
извлекаются, лишнего не отрезано. 311 тестов под `--features code` (302 без), clippy/fmt

## 2026-06-18 — `init`: создавать AGENTS.md, если файла-инструкции нет

**Что.** По фидбеку: шаг [3/4] раньше при отсутствии любого файла-инструкции просто писал
«пропускаю». Теперь, если в корне НЕТ ни одного entry-point файла (CLAUDE/AGENTS/AGENT/GEMINI/
.cursorrules/copilot-instructions), `init` СОЗДАЁТ `AGENTS.md` = `# AGENTS`-заголовок
(`AGENTS_HEADER`) + pointer-блок, чтобы у модели был вход. Если есть хотя бы один — привязываем
его, AGENTS.md не создаём (выбор пользователя «только если ни одного нет»).

Реализация — `init.rs`: `wire_entry_points` теперь возвращает `WireResult { wired, any_existed }`
(а не просто Vec) — `any_existed` отличает «файлов нет» от «есть, но блок уже актуален» (иначе
идемпотентный повтор ложно создал бы AGENTS.md). Новый `create_agents_md` (создаёт ТОЛЬКО если
файла нет — никогда не клоберит). Ветка в `run`: wired→«привязано», any_existed→«указатель уже на
месте», иначе→create_agents_md. `unwire_entry_points` (deinit): если после снятия блока остался
ровно `AGENTS_HEADER` — удаляет файл целиком (`{"removed":…}`), т.е. init→deinit round-trip
fresh-проекта в ничто; пользовательский файл с реальным контентом сохраняется (блок снят).
`create_agents_md` печатает `{"wired":…,"new_file":true}` (отдельный ключ от финального
`{"created":<папка>}`).

Контракт: help::HELP (блок init), README (врезка auto-wiring). Тесты (3 новых):
`init_creates_agents_md_when_no_instruction_file`, `init_does_not_create_agents_md_when_claude_present`,
`unwire_removes_cm_created_agents_md_but_keeps_user_file`. Поправлен integration-тест
`init_stdout_json_and_stderr_pointer` — stdout теперь JSONL (может быть строка {"wired"} перед
сводкой), парсим через parse_jsonl и берём строку с "created". 317 под `--features code` (308 без),
clippy/fmt чисто. Вживую: пустой проект → создан AGENTS.md с заголовком+блоком; с CLAUDE.md →
AGENTS.md не создан; deinit → cm-созданный AGENTS.md удалён, src/main.rs цел.

**Почему важно.** Fresh-проект без инструкций раньше не получал никакого указателя на cm — модель
не знала про инструмент. Теперь у любого проекта после init есть entry-point, при этом мы не
навязываем второй файл, если у проекта уже свой (CLAUDE.md), и чисто откатываемся в deinit.

## 2026-06-18 — Раскладка init: cm + config в корне, данные в memory/

**Что.** По запросу пользователя `init` теперь разносит раскладку: `cm(.exe)` + `config.json`
остаются в КОРНЕ проекта (куда положил пользователь), а данные (store.db, notes/, imports/,
models/) — в подпапке `memory/` (имя из `--name`, дефолт memory). Связь — поле `Config.data_dir`
(путь к данным относительно config.json, или абсолютный). Бинарь в корне читает config рядом с
собой → находит данные без `--dir`.

Реализация:
- `Config.data_dir: String` (#[serde(default="default_data_dir")] = "."). Дефолт "." = legacy
  single-folder (старые конфиги без поля работают как раньше). `init` пишет data_dir=<name>.
- `Ctx::new(dir)`: config_path=dir/config.json; data_dir читается из config (best-effort) →
  `resolve_data_dir(dir, &cfg.data_dir)`; если config нечитаем → fallback dir сам (legacy).
  store/notes/imports — под data_dir. Новый `commands::resolve_data_dir` (pub(crate)): ""/"." →
  config_dir; абсолютный → как есть; относительный → join.
- `init::run`: разнёс `data_folder = target/<name>` (данные) и `target` (config+бинарь). config
  пишется в target/config.json; cfg.data_dir=name; .gitignore — внутри data_folder (без строк
  бинаря, т.к. он в корне); бинарь копируется в КОРЕНЬ (skip если уже там). already_exists
  проверяет data_folder. Шаг [1/4]: «корень: config.json + cm; данные: <name>/».
- `deinit`: config.json удаляется из КОРНЯ (вынесен из DERIVED_FILES, отдельная ветка target+
  folder); data_folder резолвится из root config (data_dir) или --name (дефолт memory).
- `config` команда: проверяет наличие config.json ДО require_store (иначе при пропаже config
  «no memory store» вместо ясного «config.json not found» — config держит путь к данным).

Контракт: help (WHERE FILES LIVE + блок init переписаны), README (диаграмма раскладки,
Quickstart, строки init/deinit). Тесты: новые `resolve_data_dir_*`, `ctx_splits_*`,
`ctx_legacy_single_folder_when_no_config`, `deinit_finds_data_folder_via_config_without_name`;
обновлены init/deinit/lifecycle/config/import_export/api_provider тесты + common helpers
(`memory_dir`→корень, новый `data_dir`). 321 под `--features code` (312 без), clippy/fmt чисто.
Вживую: split-раскладка, работа из подпапки (binary-folder fallback), deinit чистит обе локации,
СТАРЫЙ репо-стор (memory/ без data_dir) работает без изменений.

**Почему важно (запрос пользователя).** Пользователь хотел: бинарь+конфиг там, куда положил, а
БД и прочее по умолчанию в memory/. Раньше init складывал ВСЁ в memory/ (включая бинарь). Теперь
корень чистый (cm+config), данные изолированы в memory/, и всё это с полной обратной
совместимостью со старыми single-folder сторами.

## 2026-06-18 — Ревью изменений сессии: 6 фиксов + дедуп walk

**Что.** Мультиагентное ревью (8 finder-углов → дедуп → адверсариальная верификация) накопленных
за сессию изменений (граф-кода, init-UX, split-layout). Подтверждено и исправлено 6 находок:

1. **gitignore бинаря (init.rs).** В split-layout бинарь копируется в КОРЕНЬ, а data-folder
   .gitignore больше не содержал cm/cm.exe → ~21MB exe мог попасть в коммит. Фикс: новый
   `ensure_binary_ignored(target)` — идемпотентно дописывает cm/cm.exe в КОРНЕВОЙ .gitignore
   (создаёт при отсутствии, не клоберит, маркер GITIGNORE_MARKER; no-op если юзер сам игнорит
   cm.exe). config.json рядом НЕ игнорируется (мал, коммитится).

   папке, но split-data-folder config.json не содержит (он в корне) → второй/вложенный split-стор
   не распознавался, его imports/ ре-абсорбился. Фикс: распознаёт `store.db && (imports/ ||
   config.json)`. Старый тест `prunes_any_memory_folder` покрывал только legacy-маркер (stale) —

3. **Ctx::new глотал битый config (commands.rs).** `.ok()` при невалидном (но существующем)

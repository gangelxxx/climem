# Как устроена память проекта и как сделать такую же

Это **мета-документ**: он объясняет другой модели (или человеку), как устроена папка
`memory/` в climem, по каким принципам ведётся и как воссоздать такую же систему **с нуля** в
любом проекте. Конкретные правила ведения — в [RULES.md](RULES.md); здесь — про идею,
структуру и порядок создания. (Заведено по образцу одноимённого документа из проекта ccDoc,
адаптировано под одиночный Rust-бинарник.)

> Коротко: `memory/` — это **версионируемый источник правды о проекте**, заменяющий «память
> между сессиями». Цель — чтобы новая сессия модели за минуты поняла проект и могла **быстро
> починить баг или внедрить фичу**, не переоткрывая уже известное.

---

## 1. Зачем это нужно (проблема)

Модель не помнит прошлые сессии. Без внешней памяти каждая новая сессия заново читает
исходники, переоткрывает архитектурные решения, повторяет ошибки и забывает договорённости.
`CLAUDE.md` помогает, но он один, локальный и быстро разрастается.

`memory/` решает это так:

- **Версионируется** → знание переживает сессии, видно в истории, шарится с командой.
- **Тонко, точно, живо** → не свалка; короткие файлы, один факт — в одном месте.
- **Не дублирует код/спеку** → хранит то, чего в коде нет: *почему*, инварианты, межслойную
  склейку, разборы багов, состояние задачи. Концепцию хранит `desc.md`, контракт — `help`.

Три свойства, которые надо защищать: **тонкая** (не растёт мусором), **точная** (не врёт про
текущий код), **живая** (обновляется в том же изменении, что и код).

## 2. Структура (как сейчас в climem)

```
memory/
├── README.md             # индекс: что где лежит (точка входа)
├── RULES.md              # регламент ВЕДЕНИЯ памяти (что/когда/как писать)
├── HOWTO-BUILD-MEMORY.md # этот файл: как воссоздать систему с нуля
│
│   ── «живые» файлы (правятся часто, по триггерам) ──
├── overview.md           # что за проект, стек, версии, фичи сборки, активная задача
├── structure.md          # карта модулей src/, модель данных (SQLite), потоки recall/import/embed
├── build.md              # сборка, фичи cargo, профиль, тулчейн, дистрибуция через init
├── conventions.md        # грабли / gotchas / неинтуитивные зависимости
├── known-issues.md       # баги: открытые (что чинить) + закрытые (как починили)
├── roadmap.md            # точки расширения и планы (MCP, нейромодель, PDF)
├── journal.md            # журнал сессий: что сделано и ПОЧЕМУ
│
│   ── подпапки (каждая со своим README-индексом) ──
├── rules/                # правила работы над КОДОМ проекта (не над памятью!)
│   ├── README.md
│   ├── writing-code.md   #   процесс: изучить → план → код → проверка → обновить память
│   └── code-style.md     #   оформление Rust-кода
│
├── howto/                # рецепты сквозных изменений (межслойная склейка)
│   ├── README.md
│   ├── add-command.md
│   ├── add-embedding-provider.md
│   └── add-import-format.md
│
└── notes/                # заметки по фичам/компонентам
    ├── README.md
    └── decisions.md      #   ключевые архитектурные решения и инварианты
```

**Чего нет (намеренно, по правилу роста):**

- **`deploy.md`** — у climem нет серверного деплоя; «дистрибуция» = self-copy через `init`,
  описана в [build.md](build.md).
- **`analysis/`** — заведётся, когда появится первый глубокий разбор/большой план.

### Два класса файлов

- **Индексы** (`README.md` в каждой папке) — только таблица «файл → о чём» + правило, куда
  класть новое. В индексе **нет контента**, только навигация.
- **Контент** — всё остальное. Делится на:
  - **«Живые»** (корень `memory/`): отражают *текущее* состояние, правятся по триггерам.
  - **«Накопительные»** (`known-issues`, `journal`, будущий `analysis/`, `notes/`): растут
    записями; старое не стирается, а помечается «закрыто/устарело».

## 3. Роль каждого файла

| Файл | Что хранит | Чего НЕ хранит |
|------|-----------|----------------|
| `README.md` | Индекс-таблица + связь с другими хранилищами | Контент |
| `RULES.md` | Регламент ведения памяти (триггеры, стиль, антипаттерны) | Знание о проекте |
| `overview.md` | Что за проект, стек, версии crate'ов/схемы, активная задача. **Единственный источник версий.** Сверху — «последняя ревизия: дата» | Карту модулей (она в structure) |
| `structure.md` | Карта модулей `src/`, модель данных (SQLite), потоки recall/import/embed, конфиг | Версии (они в overview) |
| `build.md` | Команды сборки, фичи cargo, профиль, тулчейн, дистрибуция | — |
| `known-issues.md` | Баги: «Открытые» + «Закрытые» (как починили + ссылка) | Полный разбор (он в `analysis/`) |
| `conventions.md` | Грабли: stdin-тело, JSONL, `/флаг`, дрейф эмбеддера, FTS, секреты | То, что очевидно из кода |
| `roadmap.md` | Точки расширения (MCP, нейромодель, PDF) и идеи | — |
| `journal.md` | По сессиям: что сделано и **почему**, что осталось | Пересказ диффа (он в git) |
| `rules/*` | Как работать над **кодом** проекта | Как вести память (это в RULES.md) |
| `howto/*` | Плейбуки сквозных задач — кодируют межслойную склейку | Теорию |
| `notes/*` | Заметки по фичам + `decisions.md` (решения/инварианты) | Сырые дампы |

Важное разделение: **`RULES.md` про память, `rules/` про код.** В обоих README — предупреждение.

## 4. Как создать с нуля (порядок шагов)

Не пиши всё сразу. Память растёт вместе с пониманием проекта. Порядок:

1. **`README.md`** — индекс с (пока пустой) таблицей. Точка входа.
2. **`RULES.md`** — регламент: карта «что куда», триггеры обновления, правила письма, чего
   не делать. «Конституция» — без неё память загниёт.
3. **`overview.md`** — самый ценный первый контентный файл. Изучи проект (корневой `README`,
   `Cargo.toml`/`package.json`, точки входа, `desc.md`-подобную спеку) и запиши: что это,
   стек, версии, как собирается, текущая активная задача. Сверху — «последняя ревизия: дата».
4. **`structure.md`** — карта модулей/слоёв, модель данных, ключевые потоки. Ссылайся на код
   по **пути + имени символа**, не по строкам.
5. **`build.md`** — команды, фичи, профиль, тулчейн (включая грабли вроде PATH к cargo).
6. **Накопительные файлы** заводи **по событию**, не заранее: нашёл баг → `known-issues.md`;
   принял решение → `notes/decisions.md`; напоролся на грабли → `conventions.md`; сделал
   нетривиальную сквозную задачу → рецепт в `howto/`; глубоко разобрал инцидент/составил
   большой план → файл в `analysis/`; завершил сессию → `journal.md`.
7. **Подпапки** (`rules/`, `howto/`, `notes/`, `analysis/`) создавай с **`README.md`-индексом**
   в момент появления первого файла внутри.
8. **CLAUDE.md** (см. §7) — тонкий: язык + инструкции агенту + ссылки сюда, и фраза
   «durable-инфа живёт в `memory/`, читать в первую очередь».

> Правило роста: **файл заводится, когда есть что в него положить**, не «про запас».

## 5. Жизненный цикл знания

```
планирую расширение → roadmap.md (строка) / analysis/<topic>-plan.md (если план большой)
делаю               → notes/<тема>.md по ходу
разобрался глубоко  → analysis/<topic>.md
принял решение      → notes/decisions.md
нашёл/починил баг   → known-issues.md (Открытые → Закрытые)
завершил сессию     → journal.md
```

Одно знание часто живёт на двух уровнях детализации: **выжимка** в живом файле
(`known-issues.md`, `decisions.md`) + **полная версия** в `analysis/`, связанные ссылкой.
Это не дублирование — это уровни.

## 6. Соглашения по письму

- **Кратко и плотно**, единый технический стиль. Без воды и пересказа очевидного.
- **Один факт — одно место.** Не дублируй между файлами — **ссылайся** (`[текст](путь)`).
  Версии — только в `overview.md`.
- **Даты абсолютные** (`2026-06-17`), не «вчера/недавно».
- **Ссылки на код — путь + имя символа** (функция/тип/модуль), не номер строки.
- **Перед добавлением проверь**, нет ли уже файла/раздела про это → **обнови**, не плоди дубль.
- **Линкуй обильно** между файлами памяти — навигация важнее изоляции.
- Язык контента — русский (как `desc.md`/`README.md` проекта); **комментарии в коде — английский**.

### Антипаттерны

- ❌ Дублировать то, что уже в **коде, `desc.md`, корневом README, `help::HELP`** — ссылайся.
- ❌ Плодить файлы ради полноты. Память — не второй архив.
- ❌ Хранить **секреты**. У climem ключ API и так не в репо (только имя env-переменной).
- ❌ Оставлять заведомо устаревшее: правишь факт в коде — правь и память тем же изменением.
- ❌ Превращать `journal.md` в пересказ диффа — пиши **решения и почему**.

## 7. Связь с другими хранилищами

- **`desc.md`** (корень репо) — design-спека, первоисточник «почему так». `notes/decisions.md`
  — выжимка со ссылками на её разделы. Не переписывай спеку в память — ссылайся.
- **Корневой `README.md`** — пользовательская документация (контракт, быстрый старт).
- **`CLAUDE.md`** (если появится; gitignored, локальный) — тонкий: язык + инструкции агенту +
  ссылки сюда. Durable-инфа живёт в `memory/`.
- **Авто-память** `~/.claude/projects/.../memory/` (per-user, вне репо) — рабочий черновик
  модели с сырыми разборами. Выжимка дистиллируется в `known-issues.md` / `notes/`.

Принцип: **версионируемое и долговечное → `memory/`; концепция → `desc.md`; контракт →
`help`; локальное/черновое → CLAUDE.md / авто-память.**

---

## 8. Поддержание свежести (анти-гниение)

- В «живых» файлах (`overview.md`) — строка «**последняя ревизия: дата**».
- Раз в N времени — быстрый проход «что устарело» (особенно версии и активная задача).
- Когда проект встанет под git — добавь **Stop-хук**, напоминающий обновить память, если
  менялся код, а `memory/` — нет. Это превращает «обновляй память» в часть процесса.
- Факт/рецепт, разошедшийся с кодом, — **чини сразу** (часть работы, не долг).

## 9. Чек-лист «память сделана правильно»

- [x] Есть `README.md`-индекс — за один файл понятно, что где.
- [x] Есть `RULES.md` — правила ведения зафиксированы.
- [x] `overview.md` отвечает «что это и в каком состоянии» за минуту чтения.
- [x] Версии/даты — в одном месте, абсолютные, свежие.
- [x] Нет дублей: повторяющийся факт заменён ссылкой.
- [x] Нет секретов.
- [x] Баги имеют статус (на момент заведения — открытых нет).
- [x] `journal.md` объясняет *почему*, а не *что в диффе*.
- [ ] Память правится в том же коммите, что и код (включить, когда проект встанет под git;
      идеально — с хук-напоминанием).

# Архитектурные решения и инварианты

Выжимка «почему так». Полное обоснование концепции — в [`../desc.md`](../desc.md) (ссылки
вида `§N`). Здесь — короткие тезисы с указанием, где это держится в коде, чтобы решение не
переоткрывали и случайно не сломали.

## Память как исполняемый инструмент, а не md-файл (`desc.md §1–2`)

Данные на диске (SQLite), вне окна контекста; модель дотягивается по запросу. Следствие для
всего кода: **не тянуть весь архив в вывод**, отдавать релевантный срез (`recall`).

## Короткоживущий процесс (`desc.md §9`)

Каждый вызов: открыл store/config → одна операция → JSONL → выход (`commands::Ctx::open`,
`main::run`). **Инвариант:** не держим долгоживущее состояние, нет «сессии». Падать в
середине сессии нечему — это надёжность по дизайну. Не добавляй фоновых демонов/кэшей в
процессе без веской причины.

## Один файл `store.db` + rollback-журнал (`desc.md §11`)

`store.db` — **один** файл (по-прежнему), поэтому **rollback-журнал, не WAL**
(`store::Store::open`): WAL породил бы `-wal`/`-shm` сайдкары и второй+третий файлы.
Инвариант защищается и в `.gitignore` (исключает `*.db-wal`/`*.db-shm`).

> ⚠️ **Переосмыслено (2026-06-17):** «один файл» теперь не значит «источник правды».
> `store.db` **производный и одноразовый** (правда — в `notes/`+`imports/`, см. решение ниже).
> Рационал «один копируемый файл» сохраняется, но он про удобство индекса, а не про
> сохранность памяти: коммитится правда, не БД.

## md-файлы — источник правды, `store.db` — производный кэш + граф (2026-06-17, `desc.md §3`)

**Было:** `store.db` — единственный источник правды (старый инвариант «один портативный файл
store.db = правда»). **Стало:** правда — файлы на диске; `store.db` целиком пересобирается из
них (`commands::reindex`). Зачем: контент читаем, грепаем, правится руками и ревьюится в PR;
сбой/смена эмбеддера/размерности не теряет память. Реализовано модулями `note.rs` (формат md)
и `graph.rs` (граф) + командами `reindex`/`related`; `config.version` бампнут 1→2.

**Инварианты (новые/изменённые):**

- **(a) md = правда, store.db = производное/пересобираемое.** `notes/<id>.md` (1 файл = 1
  заметка) + `imports/` (оригиналы + `.meta.json`-сайдкары) — правда; индекс восстановим
  `reindex`. Старый инвариант «store.db = правда» **отменён**.
- **(b) Запись файла — точка коммита, индексация — best-effort.** `remember` пишет
  `notes/<id>.md` ДО эмбеддинга и **не падает на кривом эмбеддере** (note сохранён, `{"id"}`
  напечатан, индекс чинится `reindex`; `commands::index_note_best_effort`). `import`
  копирует оригинал в `imports/` до нарезки.
- **(c) Производные таблицы стираются+пересобираются; `oplog`+`meta` переживают.**
  `store::wipe_derived` чистит `notes`/`notes_fts`/`sync`/`imports`/`edges`; `oplog`
  (история) и `meta` (`embedder_signature`) — не реконструируемы, поэтому остаются.
- **(d) Всё, что знает индекс/граф, должно быть выразимо в md** (frontmatter
  `slug`/`relations`/`tags`/`source` + body `[[ссылки]]`) **+ сайдкары импортов** — иначе
  появятся два источника правды и БД перестанет быть одноразовой.
- **(e) Чанки — контент-адресные и не удаляются напрямую.** id чанка =
  `content_hash(file_hash#n)` (стабилен между пересборками); `forget` на `kind=chunk`
  **отказывает** с подсказкой (правь `imports/` + `reindex`).
- **(f) Граф рёбер выводится из md (bulk), висячее — first-class.** Цель разрешается по
  `slug` (по умолчанию) или префиксом `id:`; неразрешённое — висячее ребро (`dst_id NULL`),
  оживает на следующем `reindex`.
- **(g) `store.db` — по-прежнему один rollback-журнальный файл**, просто теперь одноразовый
  (см. решение выше).
- **(h) Удаление/прунинг заметки переводит ВХОДЯЩИЕ рёбра в висячие** (2026-06-18, закрыло
  B2/B3). `forget` и prune в `reindex` зовут не только `delete_edges_from` (исходящие), но и
  `store::dangle_edges_to` (`UPDATE edges SET dst_id=NULL WHERE dst_id=?`). Иначе входящее
  `A→B` осталось бы «разрешённым» на мёртвый id (тёмная материя, не выводимая из md) и могло
  бы воскреснуть на чужую заметку при переиспользовании id. `dst_raw` (дословная цель из md A)
  сохраняется, поэтому ребро корректно ре-резолвится по (h+i).
- **(i) Инкрементальный `reindex` оживляет висячие рёбра даже у НЕизменившихся источников**
  (2026-06-18, закрыло B1/L3). Второй проход, помимо `changed_notes`, переиндексирует рёбра
  всех `store::dangling_edge_sources()` (`DISTINCT src_id WHERE dst_id IS NULL`), перечитывая
  их md с диска. Нужно потому, что новую цель мог добавить сам `remember` — тогда `reindex`
  видит `{"changed":0}`, но множество резолвимых slug/id выросло. Проход идемпотентен и почти
  бесплатен (пустой список при отсутствии висячих).
- **(j) Граф направленный; предикаты нормализуются.** Рёбра однонаправленные (`A→B` = «A
  ссылается/зависит от B»): `related`/`recall --related` идут вперёд по `edges_from`,
  `backlinks` — назад по `edges_to` (использует индекс `edges_dst`). Предикаты проходят через
  `graph::normalize_predicate` (lowercase + схлопывание `-`/`_`/пробелов в `_`), так что
  `depends_on`≡`depends-on`≡`Depends On`; фильтр `--predicate` нормализуется так же. Рёбра
  дедупятся по `(predicate, resolved-key)` в `index_note_edges` ДО вставки (две записи одной
  цели разным написанием → одно ребро); PK по `dst_raw` не трогаем (он защищает
  воспроизводимость wipe+rebuild).

**Сопутствующие решения:**
- **id заметки — короткий lowercase-hex** (`store::mint_hex` — FNV-fold time+pid+counter, с
  retry на коллизию), авторится во frontmatter, `id == notes/<id>.md`. Весь код `i64→String`;
  `commands::parse_id` валидирует `^[0-9a-f]+$`. Почему не autoincrement: id живёт в md
  (правде), должен быть стабилен между пересборками и не зависеть от порядка вставки в БД.
- **content_hash = FNV-1a** (`util::content_hash_hex`) — общий для `sync` change-detection и
  контент-адресных id чанков; не криптография, нужен дешёвый стабильный хэш.
- **Разрешение цели графа — slug-или-`id:`, без hex-эвристики** (`graph::resolve_target`):
  slug вида `db-schema` не должен случайно совпасть с id; явный `id:` — единственный способ
  адресовать по id.
- **`normalize_slug` сохраняет кириллицу** (НЕ FTS-диакритик-фолдинг): контент мультиязычный,
  человеко-имена графа русские.

## Гибридный recall скрыт за одной командой (`desc.md §3, §11`)

FTS5 (ключевые слова) + векторный косинус (семантика/морфология), слитые в одну выдачу
(`search::recall_with`). **Решение:** гибридность — деталь реализации, контракт остаётся
`recall "<query>"`. Веса настраиваются конфигом (`search.hybrid_weights`), не пересборкой.

## Слияние каналов — RRF, не per-channel min-max (token-efficiency-plan R1, 2026-06-18)

**Было:** каждый канал нормировался min-max в [0,1] и складывался. bm25 (безграничный) и
косинус ([-1,1]) плохо сопоставимы → ранжирование шумело, нельзя было уверенно резать `limit`.
**Стало:** Reciprocal Rank Fusion — `score = w_fts/(k+rank_fts) + w_vec/(k+rank_vec)`,
`k = search.rrf_k` (60), ранги — позиции в сортировке каждого канала. Сливаются **позиции**, а
не несопоставимые величины → стабильный top-k, который безопасно усечь до маленького `limit`
(дефолт 5). **Следствия/инварианты:**
- Отчётные `score/fts/vector` — это **вклады RRF** (мелкие, `~w/(k+rank)`, НЕ [0,1]);
  `score == fts + vector`. `--min-score` задаётся в этой шкале. Старого инварианта «score в
  [0,1]» больше нет (поправлено в [../conventions.md](../conventions.md)).
- Порядок выдачи **детерминирован**: кандидаты из `BTreeSet`, сортировки с tie-break по `id`
  (закрыло баг tie-order, [../known-issues.md](../known-issues.md)).
- Чистый Rust, без новых зависимостей (минимальное дерево депов).

## Граф-обход — одно ядро `bfs_neighbors`; граф-канал recall настраивается (2026-06-18)

**Было:** две независимые копии BFS — `commands::related` (с предикат-фильтром и пропуском
stale) и `search::neighbor_ranks` (без них, и без `store.get` → могла ранжировать по мёртвому
`dst_id`). **Стало:** общее ядро `search::bfs_neighbors(store, start, depth, predicate,
include_dangling)` (D1): детерминированный BFS, сид `visited` стартовым узлом, **live-валидация
каждого разрешённого ребра** через `store.get` (D2 — stale-ребро не протекает ни в один тракт),
сортировка `(distance, predicate, resolved<dangling, key)`. `related` просит `include_dangling=
true` и режет `limit`; `neighbor_ranks` — `false` (канал ранжирует только живые заметки).
**Граф-канал recall** теперь читает глубину из `config.search.graph_depth` (дефолт 2, был
хардкод; D8) и опциональный `config.search.graph_predicate` (D9) — раньше глубина «2» и
отсутствие фильтра были скрытой связкой. `cm related` сохраняет свой `--depth` (дефолт 1) —
команды намеренно разные по умолчанию. **Discoverability (D3):** `recall --related` при
`hybrid_weights.graph==0` печатает stderr-хинт с готовой `config set`-командой (канал off по
умолчанию, иначе молча no-op). Поднимать дефолт `graph` не стали — это поменяло бы ранжирование
у всех.

## `backlinks` — обратная сторона направленного графа (2026-06-18, D6/D7)

Граф направленный (инвариант (j)), и до сих пор не было способа спросить «кто ссылается на
эту заметку». Добавлена команда `cm backlinks <id> [--predicate P] [--limit N]` →
`store::edges_to` (`WHERE dst_id=?`, ожил индекс `edges_dst`). Один хоп, только живые
источники, вывод `{"id","kind","predicate","preview"}`. Контракт синхронизирован
(`help::HELP`, README, desc.md) — флаги `--predicate`/`--limit` уже в `VALUE_FLAGS`.

## Худой вывод `recall` + проекция полей (token-efficiency-plan E1, 2026-06-18)

`recall` по умолчанию печатает только `id, kind, body` + непустые `tags`/`origin`/`source`;
отладочные `score/fts/vector` — под `--explain`; `--fields` — whitelist `output::RECALL_FIELDS`.
**Почему:** срезает «конверт» вывода (повторяющиеся ключи, тройку чисел, null-поля, таймстемпы)
без потери качества выдачи — главный дешёвый токен-выигрыш. Контракт остаётся **JSONL на
stdout**; полная запись доступна через `get <id>`. Байт-стабильность (ключи алфавитно, без
per-call таймстемпов) — задел под prompt-caching у потребителя (плана A2).

## Breadcrumb-заголовки в чанках (token-efficiency-plan R3, 2026-06-18)

`chunk::markdown` строит полный путь заголовков по уровневому стеку: `origin` = `файл › H1 ›
H2 › H3`, путь предков префиксуется в тело чанка (вычитается из бюджета слов). **Почему:**
немного слов контекста поднимают нужный чанк дешевле, чем более крупный чанк. На `local`
выигрыш в основном лексический; семантический — под `api`.

## Char-3grams в локальном эмбеддере — ради русского (`embed/hashing.rs`)

FTS5 не стеммит: «авторизация» ≠ «авторизацию». Локальный эмбеддер хеширует **и слова, и
char-3grams** (`^слово$`), поэтому ловит общую морфему и поднимает нужную заметку. Это не
семантическая модель, а лексико-морфологическая близость — осознанный компромисс ради
офлайна и нулевых зависимостей. Знаковое FNV-1a + L2-норма держат коллизии около нуля и
делают косинус сопоставимым для коротких/длинных текстов.

## Сменный провайдер эмбеддингов за одним интерфейсом (`desc.md §3`)

`trait Embedder` (`embed/mod.rs`), выбор в `embed::build` по `config.embedding.provider`.
**Решение:** `api` включён в дефолтную сборку (фича `api`), так что переключение на
нейромодель — **рантайм-конфиг**, без пересборки. Локальная нейромодель (ONNX) добавляется
третьим провайдером в тот же интерфейс — слот заложен (`weights_path`).

## Ключ API — только из окружения (`desc.md §5`)

В `config.json` хранится лишь *имя* env-переменной (`embedding.api_key_env`), не ключ
(`ApiEmbedder::from_config` читает `std::env::var`). **Инвариант:** сырых секретов в конфиге
не бывает; `config::mask_secrets` дополнительно маскирует на выводе. Не вводи поле под ключ.

## `help` = контракт внутри бинарника (`desc.md §7`)

Текст контракта живёт в `help::HELP` и правится вместе с поведением — не может «уехать», как
отдельный doc. **Правило:** меняешь команду/флаг — правь `help::HELP` в том же изменении.

**Язык и адресат `help::HELP` — английский, под слабую LLM (2026-06-18).** Единственный
потребитель `cm help` — это модель-агент, а не русскоязычный человек, поэтому текст
**намеренно на английском** (исключение из «`desc.md`/`README.md` — русские»: они объясняют
проект человеку, `help` — инструктирует модель в рантайме). Стиль рассчитан на самую слабую
модель: буквальный язык без идиом, блок «READ THIS FIRST» с 6 правилами, таблица-навигатор
«WHICH COMMAND DO I USE?», для каждой команды — copy-paste пример и точная форма JSON-вывода.
**Правило:** не переводи `help` обратно на русский «для единообразия» — это осознанный выбор.
Pointer (`help::pointer`) при этом остаётся русским (его вставляют в человеко-читаемый
`CLAUDE.md`/системный промпт). Тест `cli_help::help_command_prints_contract` проверяет
английский заголовок `COMMANDS` (был `КОМАНДЫ`).

## Ошибки самовосстанавливаются (`desc.md §4`)

`util::AppError::with_hint` несёт короткий копируемый пример; `main` печатает его + отсылку к
`cm help`. **Правило:** новая ошибка ввода/контракта — давай `with_hint` с верным
примером, чтобы модель чинилась за один шаг.

## Тело — через stdin; вывод — JSONL (`desc.md §4`)

Снимает экранирование кавычек на входе и даёт парсимый, цепляемый выход. Детали и исключения
— в [../conventions.md](../conventions.md).

## Минимальное дерево зависимостей

Свой парсер аргументов (`cli.rs`), своё UTC-время без `chrono` (`util::iso_utc`), `ureq` с
`native-tls` (без ring/nasm на Windows). **Направление:** не тащить тяжёлые crate'ы ради
мелочей; тяжёлое (PDF) — за опциональной фичей.

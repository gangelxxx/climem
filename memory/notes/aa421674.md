init теперь идемпотентен: достраивает раскладку, не затирает (2026-06-20). Прежний guard `data_folder.exists()` → ранний already_exists заменён на проверку ПОЛНОТЫ. `init::LayoutState::probe(target, data_folder, config_path)` снимает состояние всех артефактов (config.json, store.db, notes/imports/models, data-.gitignore, бинарь в корне, root-.gitignore-покрытие cm.exe, CM_GUIDE.md) — чистые ФС-чтения до любых правок. `is_complete(target, exe_display)` = все части + `instruction_pointer_current` (хотя бы один instruction-файл несёт БАЙТ-идентичный текущему блок — устаревший путь при re-init под новым --name → incomplete → re-wire). Полная раскладка → `{"status":"already_exists"}` и выход, НО short-circuit пропускается при явном непустом `--docs` (`p.value("docs").is_some_and(docs_arg_has_entry)`): явный `--docs <paths>` импортирует доки даже в готовый store (правка крайнего случая «--docs игнорировался на готовой раскладке», 2026-06-21). Иначе тело выполняется, создавая ТОЛЬКО недостающее (каждый кусок под `if !pre.<part>`). store.db не пересоздаётся при наличии (Store::create и так идемпотентен — CREATE TABLE IF NOT EXISTS, но skip чтобы не трогать mtime). **ПЕРЕИНДЕКСАЦИЯ ДОКОВ ПРИ ПОТЕРЕ store.db (правка 2026-06-21, e2e на iscreen).** Если store создавался заново (`!pre.store`) И на диске есть md-truth (`has_md_truth`: ≥1 .md в notes/ или imports/) — init вызывает `reindex_documents(target, data_folder)` = `commands::reindex(&Parsed::parse(&[]), &Ctx::new(target))` сразу после скаффолда (best-effort, не валит init). Без этого repair после удаления store.db оставлял ПУСТОЙ индекс (переиндексировался только код в шаге [4/N]), и `recall` молчал до ручного `cm reindex`. Гейт `!pre.store && has_md_truth` точечный: свежая установка (пустые папки) и present-store не триггерят. config.json при наличии ЧИТАЕТСЯ (Config::load), не перезаписывается — флаги --model/--provider/--endpoint/--dimension на существующем конфиге игнорируются (warning в stderr); битый config.json → фатальная ошибка (не маскируем дефолтами). Финальный JSON: `status` = created (свежая) | repaired (достройка) | already_exists (полная). `any_present()` (= repairing) считает только store/dirs/data-.gitignore/guide — НЕ бинарь, НЕ root_ignored, и НЕ config (правка 2026-06-21: deinit ОСТАВЛЯЕТ config.json, поэтому одинокий config — after-state деинита, и init поверх него = свежий `created`, не repair; config всё равно переиспользуется). Шаги wiring/--docs/--code гонятся всегда (идемпотентны). Манифест на repair МЕРЖИТСЯ (read_manifest_opt): сохраняем pre-init .gitignore-снимок и created_agents_md/created_cm_guide из ПЕРВОГО init (OR-in новые), docs.extend — иначе откат deinit испортится. Контракт синхронизирован: help::HELP (блок init, IDEMPOTENT), README (врезка). Отменяет note 64ca420a «already_exists по data_folder». Связано: [[init-no-arg-defaults]].
